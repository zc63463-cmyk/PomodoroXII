### 4.2 MAJOR：4 个关联表实体墓碑缺失

**位置**：`services/relation.py:79-91`（`unlink` 方法）

**问题**：`RelationService.unlink()` 直接 `db.delete(existing)` 删除关联表行，**不创建墓碑**。这 3 个关联表（sessionQuickNote / scheduleQuickNote / taskQuickNote）均在 `ENTITY_REGISTRY` 中参与同步。后果：

- 设备 A 将快捷笔记从任务 X 取消关联 → 服务器删除 taskQuickNote 行
- 设备 B 增量 pull → **收不到该删除事件**（无墓碑）→ 本地保留陈旧关联

`memoComment` 更甚——**无任何 CRUD 路由**，只能通过 sync push 创建/删除，删除路径同样无墓碑。

**建议**：让 `RelationService` 继承或委托 `BaseService` 风格的墓碑机制；或为关联表删除补充 `TombstoneService.create()` 调用。

### 5.1 严重度汇总（本次独立复核）

| 级别 | 数量 | 关键项 |
|------|------|--------|
| CRITICAL | 0 | —（旧报告的 3 项均已修复） |
| MAJOR | 5 | 关联表墓碑缺失 / 无 frontmatter / 无限流 / 无安全头 / delete_folder TOCTOU |
| MINOR | 7 | 弱密钥黑名单缺口 / list_trash 内存分页 / file_system 无连接池 / 内联 Service 位置 / lint 非阻塞 等 |

### 5.2 MAJOR 项详情

**M1 — 认证端点无限流**：`/api/v1/auth/login` 无 IP 级速率限制，可被暴力破解。建议引入 `slowapi`。

**M2 — 安全响应头缺失**：`middleware.py` 仅有 `RequestIdMiddleware`，缺少 `X-Content-Type-Options`、`X-Frame-Options`、`Strict-Transport-Security`、`Content-Security-Policy`。建议新增 `SecurityHeadersMiddleware`。

**M3 — delete_folder TOCTOU 竞态**（file_system 子系统）：`folder_ops.py:240-243` 在锁内收集 note_rows 后释放锁，再逐个 `delete_note`（重新加锁）。两步之间笔记可能被并发修改，导致 `KeyError` 中断递归删除。

**M4 — 无 YAML frontmatter**：见第七节详述。

**M5 — 关联表墓碑缺失**：见第四节。

### 5.3 MINOR 项

**m1 — 弱密钥黑名单缺口**：`settings.py:76` 黑名单为 `{"change-me", "change-me-in-production", "secret", "password"}`，但 `.env` 实际使用 `dev-test-secret-key-for-testing-only` 不在列。虽 environment=development 不会硬失败，但若该值被误带入生产可绕过校验。建议改为"生产环境强制密钥长度 ≥32 且非已知弱值"。

**m2 — list_trash 内存分页**：`trash.py:52-123` 将所有 trashed items 加载到 Python list 再切片分页。墓碑累积（90 天 TTL）后内存压力上升。建议改为 DB 层分页。

**m3 — file_system 无连接池**：每个操作 `sqlite3.connect()` 新建连接，高并发开销大；同一逻辑操作内多次 `_connect` 不在同一事务。

**m4 — 内联 Service 位置**：8 个 Service 类定义在路由文件内（habits/sessions/...），违反三层分离，阻碍未来 MCP Server 复用。功能正常，属架构债。

**m5 — CI lint 非阻塞**：`ci.yml:72` `ruff check ... || true`，lint 失败不阻断。建议改为阻塞或单独 job。

**m6 — CORS 配置**：`allow_credentials=True` + 显式 origin 列表（非通配）——配置正确，但开发环境默认 origin 较宽。

**m7 — status() UNION ALL 注入面**：`sync.py:531` 将 `__tablename__` 拼入 SQL text。表名来自 ORM 硬编码（非用户输入），当前安全；但属脆弱模式，若未来引入动态表名需警惕。

### 5.4 安全亮点 ✅

- PyJWT（非 python-jose）+ bcrypt 12 rounds + 算法固定 `algorithms=[settings.algorithm]` 防混淆
- 双 token 模型（master 7d / space 8h）+ space_id 实时校验防伪造
- 多空间隔离通过独立 SQLite DB 实现，无跨空间泄露
- FTS5 查询双引号转义防注入（`search_ops.py:40`）
- 参数化查询贯穿全栈，`status()` 的表名注入面已说明安全理由
- `.env` 未被 git 跟踪（已验证 `git ls-files`）
- 500 错误不泄露堆栈，仅返回通用消息 + 服务端日志
- 关联表 link 用 IntegrityError 捕获处理 TOCTOU（`relation.py:65-77`）

8-Mixin 组合模式（StorageBase + 7 OpsMixin）设计清晰：
