# TipTip × PomodoroXII 多轮 Grill Me 深度决策总纲

> 状态：当前产品语义总纲，供下一轮正式规格修订与原型审查使用  
> 日期：2026-07-10  
> 适用范围：单人 Pomodoro Space 内的 Project、WorkItem、Cycle、L3、FocusSession、WorkItemNote、待处理区与执行化身  
> 解释优先级：本总纲与 `ENTITY_ALIGNMENT_ADDENDUM.md` 晚于早期 v1/v2/v3 概念原型；发生冲突时，以本总纲为准

---

## 1. 执行摘要

这几轮 Grill Me 的最终结果，不是给原有任务系统增加几个字段，而是完成了五次关键纠偏：

1. **从“三级任务承接番茄时间”纠正为“二级承接投入，三级表达具体成果”**。
2. **从“Session 内临时行动项”纠正为“三级仍是正式 WorkItem，Note Checklist 才是行动指导”**。
3. **从“三级全部完成驱动父项复核”纠正为“二级累计投入达到预计上限才显著复核”**。
4. **从“多人负责人”纠正为“单人执行化身”，撤销成员、权限和 assignee 假设**。
5. **从“阻塞区装下一切问题”纠正为“真正阻塞、推进异常、待复盘分组治理”**。

当前稳定产品句子是：

> 用户在单人隔离 Space 内，以 Project 和最多三层 WorkItem 管理长期工作；二级 WorkItem 承担预计投入、实际 Session 投入与 Cycle 容量，三级 WorkItem 表达可确认的具体成果；Session 帮助用户在临近执行时选择或长出三级成果，并通过 Orbit 执行窗管理成果草稿、行动 Checklist 与执行化身；系统用投入上限、阻塞和多轮未完成等证据提醒复核，但不替用户判定失败或完成。

## 2. 多轮 Grill Me 的演进脉络

### 2.1 第一轮：任务空间与 Session 的边界

最初问题是：PomodoroXII 的“分类—项目—模块—子任务”是否应该直接映射为任务空间，以及 Session 中的小任务究竟是什么。

早期尝试将 Session 内拆解建模为临时行动清单，但后续被否决。用户明确要求：

- “+”创建的是正式第三级 WorkItem，不是 Session 临时待办；
- 三级可跨多个 Session 存活；
- Session 只是 25/45/60/90 分钟等时间容器，不代表一定完成任务；
- 碎碎念与行动清单不同，三级 WorkItem 可以拥有长期备注。

本轮最终形成“混合入口”：

- 已经知道具体成果：开始前选择或新建三级 WorkItem；
- 暂时不知道：允许只对二级开始；
- 执行中发现：可新建三级；
- 结束后才理解成果：可补建三级；
- 第三级是推荐成果拆解，但不是开始 Session 的门槛。

### 2.2 第二轮：L2/L3 空间理念与 UI 探索

原型先后经历：

1. v1 单中心轨道；
2. v2 多中心生命线；
3. v3 纯 2D 思维导图式拆解。

稳定下来的不是圆球尺寸、生命线或动画，而是产品理念：

> **L3 像思维导图一样发散拆解你的任务。**

稳定语义包括：

- L2 主要表达 Project 与一级/二级 WorkItem；
- L3 主要表达二级 WorkItem 与三级 WorkItem 的成果拆解；
- 展开/折叠只是父子事实的空间投影；
- “吐出”表示识别或展开成果，“吃回”表示收纳到成果与历史；
- UI 动画不能反向定义创建、删除或状态规则。

### 2.3 第三轮：WorkItem 与 WorkItemNote 实体

通过 Grill Me 确认：

- 一级、二级、三级是同一种 `WorkItem`；
- 层级由 `parentId` 深度派生，不保存永久 `level` 类型；
- 最大三层，三级不能直接创建第四级 WorkItem；
- 三级继续拆解时提供双出口：添加 Note，或人工提升后再拆；
- 不自动提升，避免隐式重组任务树。

`description` 与 `WorkItemNote` 分工：

| 对象 | 回答的问题 | 内容 |
|---|---|---|
| `description` | 是什么、为什么做、完成结果是什么 | 背景、目标、范围、完成定义 |
| `WorkItemNote` | 怎么推进、注意什么 | 步骤、行动指导、约束、注释、上下文 |

WorkItemNote 首版采用“一项一文档 + 稳定 Block”：

- `paragraph`
- `heading`
- `ordered_list`
- `unordered_list`
- `checklist`

列表最多两层。Checklist 勾选是共享、跨设备同步、永久保留的 Note 内容事实，但不参与 WorkItem 状态、进度、容量、风险、Cycle 或 Session 成果。

Note 列表项提升为 WorkItem 时：

- 源 WorkItem 未到三级：创建其子项；
- 源为三级：创建到同一二级父项下，成为三级同级项；
- 原 Note 列表项转成新 WorkItem 引用，不再维护独立 Checklist 状态。

### 2.4 第四轮：Session、投入和三级成果

核心纠偏是：**Session 时间不分给三级 WorkItem。**

当前模型：

```text
二级 WorkItem：投入与容量容器
├── 三级 WorkItem A：本轮成果
├── 三级 WorkItem B：本轮成果
└── FocusSession：净专注时长全部归二级
```

稳定规则：

- 一个 Session 只能归属一个二级 WorkItem；
- 本轮三级项必须属于同一二级；
- Session 可以没有三级计划；
- 三级计划可在开始前、执行中或结束后新增；
- 加入本轮计划不自动改变三级状态；
- 新建三级默认未开始；
- 三级允许在结束确认时从未开始直接完成，不伪造“进行中”；
- 未勾选三级默认保持原状态，不自动结转；
- Session 对三级保存稳定 `workItemId` 和当时标题、父项、顺序等快照；
- 三级不拥有由 Session 分配的实际分钟。

二级容量口径：

- 二级预计投入覆盖完成该成果所需的全部工作；
- 天然包含已拆出和尚未拆出的三级成果；
- 三级默认不估时、不独立加入 Cycle、不单独选择 Committed/Planned/Stretch；
- 用户主动给三级填写估时，只作局部参考，不叠加 Cycle 容量；
- 三级只展示状态计数，不换算为父项完成百分比。

### 2.5 第五轮：复核、待处理区、单人空间与执行化身

本轮修正了四个旧假设：

#### 父项复核

- 三级全部完成只形成成果摘要；
- 不触发父项复核弹窗；
- 二级投入低于预计下限：不提示；
- 进入预计区间：仅静态提示；
- 达到或超过预计上限：显著复核；
- 复核可选择完成二级、更新剩余估算、标记范围扩大、处理阻塞或延后到复盘点；
- 复核不阻止 Session 结束。

#### 待处理区

统一入口分成三组：

1. 真正阻塞；
2. 推进异常；
3. Session 待复盘。

三者不能共用同一状态。

#### 单人 Space

PomodoroXII 是单用户、多隔离 Space 产品，而非团队 Workspace：

```text
Pomodoro Space
└── Project Group（可选轻量分类）
    └── Project
        └── WorkItem
```

撤销成员、邀请、负责人、权限、成员容量和跨负责人确认。

#### 执行化身

原“负责人”展示位迁移为执行化身。牛马、小猪、哈吉米、齐天大圣属于 `Session × 三级 WorkItem` 的本轮实际执行模式，不是永久 WorkItem 属性。

## 3. 当前权威领域模型

### 3.1 核心实体

```text
PomodoroSpace
└── ProjectGroup? 
    └── Project
        └── WorkItem（最多三层）
            ├── description
            ├── WorkItemNote
            │   └── NoteBlock / ListItem / ChecklistItem
            └── Relation

Cycle
└── 二级 WorkItem Membership

FocusSession
├── 二级 WorkItem 归属
├── SessionWorkItemPlan × N
│   └── 同一二级下的三级 WorkItem 引用 + 快照
├── SessionWorkItemOutcome × N
├── Session 随记
└── SessionReview
```

### 3.2 事实源矩阵

| 事实 | 唯一事实源 | 说明 |
|---|---|---|
| WorkItem 结构与状态 | 任务空间 | PomodoroXII 不维护可写副本 |
| WorkItemNote | 任务空间 | Orbit 读取同一事实投影 |
| Cycle 与容量 | 任务空间 | 二级 WorkItem 是容量口径 |
| Session 时间与行为 | PomodoroXII | 包括暂停、中断、mood、注意力等 |
| Session 的二级归属 | PomodoroXII | 记录当时引用和快照 |
| 本轮三级计划/结果 | PomodoroXII | 不复制 WorkItem 当前状态 |
| 完成/取消三级命令 | 任务空间执行 | PomodoroXII 记录命令结果，可重试 |
| 待处理区 | 派生投影 | 来自 WorkItem、Session、Cycle 事实，不是新状态仓库 |

### 3.3 核心不变量

1. 一个 Session 只有一个二级 WorkItem 归属。
2. 一个 Session 的三级计划项必须属于该二级。
3. Session 净专注时长只计入二级，不分配给三级。
4. 三级项加入本轮计划不自动变为进行中。
5. 三级项可从未开始直接完成，历史记录来源 Session。
6. 三级全部完成不自动完成二级，也不触发显著复核。
7. 二级预计投入是包含三级成果的完整估算，不与三级参考估时相加。
8. 三级不独立加入 Cycle，也不单独承担三档角色。
9. WorkItemNote Checklist 不进入 Session 成果清单。
10. 计时中的成果完成先作为可撤销草稿，Session 结束时统一提交。
11. Session 时间更正不自动回滚 WorkItem 成果状态。
12. 历史 Session 使用当时结构快照，不随后续 WorkItem 重组重算；Session 运行中二级或三级发生移动、删除、取消或父项变化时，本轮归属和计划快照保持冻结。结束提交前逐项校验：仍存在且状态兼容时可提交；已删除/取消/父项不再匹配时只保留 Session 历史结果并进入冲突处理，不自动移动、复活或覆盖 WorkItem。
13. 二级完成时若仍有活动三级项，不能直接留下可执行悬挂项：用户必须选择取消、移动到其他未完成二级、重新打开原二级后继续，或返回；纯历史完成/取消项可以保留在成果区。
14. 所有数据和趋势限制在当前 Pomodoro Space 内；Project、WorkItem、Cycle、FocusSession 及其跨域关联必须携带同一个 `spaceId`，任何跨 Space 引用、移动、聚合或命令均拒绝。
15. Space 归档后业务事实只读保留；删除 Space 必须走 PomodoroXII 的空间级显式删除流程，任务子系统不得单独删除或迁移其中数据。
16. 执行化身不决定成果结果，不形成效率排名。

## 4. 完整用户流程

### 4.1 开始前

用户从不同层级进入：

| 入口 | Session 二级归属 | 三级计划 |
|---|---|---|
| 从三级启动 | 自动取其二级父项 | 自动加入该三级 |
| 从二级启动 | 当前二级 | 可选择、新建多个三级，也可空计划 |
| 从一级启动 | 必须先选择或新建二级 | 再决定是否拆三级 |

批量新建三级只要求标题。description、Note、关系等可稍后补充。

### 4.2 计时中

Orbit 可从 L3 桌宠变身为执行面板：

```text
当前二级 WorkItem

本轮成果清单（正式三级 WorkItem）
- 设为当前项
- 打亮钩
- 从本轮移除
- 本项不再需要

当前三级 WorkItem
└── 行动清单（WorkItemNote Checklist）

执行化身信号
- 牛马 / 小猪 / 哈吉米 / 齐天大圣
```

操作语义：

- 设为当前项：只切换执行上下文；
- 打亮钩：保存可撤销的本轮完成草稿；
- 从本轮移除：只修改 Session 计划；
- 本项不再需要：二次确认，结束时提交取消命令；
- Note Checklist 勾选：只更新 Note 内容；
- 执行化身：写到当前三级项的本轮 Outcome 草稿；
- 不记录三级分钟分配。

### 4.3 结束后

Session 结束分为五组信息：

| 信息组 | 目的 | 默认要求 |
|---|---|---|
| 有效性 | 是否计入二级实际投入 | 正常结束默认有效 |
| 三级成果结果 | 完成、有推进、卡住、未触达、取消 | 根据操作预填，可修正；取消需二次确认 |
| 总体推进评价 | 本轮对二级工作块的整体效果 | 选填 |
| Mood | 用户当时的个人情绪 | 选填 |
| 逐三级执行化身 | 具体成果的主导执行模式 | 选填、轻提醒 |

提交顺序：

1. 先保存 Session 时间事实；
2. 为每个三级结果生成稳定幂等键 `sessionId + workItemId + outcomeRevision`；
3. 分别提交三级完成/取消命令，单项成功或失败互不回滚；
4. 保存每项命令的 pending/succeeded/conflicted/failed 结果；
5. pending/failed 可按同一幂等键重试，conflicted 必须展示当前 WorkItem 状态并由用户决定；
6. 独立更新二级 EffortProjection；
7. 判断是否达到二级预计上限；
8. 必要时生成显著复核卡或进入待处理区。

部分成功时，已成功的 WorkItem 保持正式结果；失败项显示“结果已保存，任务状态待同步/待处理”，不回滚 Session，也不重复执行成功命令。系统必须提供按 Session 对账入口，直到所有命令进入 succeeded、conflicted 后已处理或用户明确放弃同步。

### 4.4 延迟复盘与离线

- Session 时间可先结束，三级成果结果可稍后补充或由用户明确选择“跳过本轮成果复盘”；总体推进、Mood、执行化身均为选填，留空本身不产生待复盘；
- “待复盘”只在存在已触达三级项且其成果结果既未确认、也未明确跳过时产生；无三级计划、全部未触达或用户明确跳过时，不产生待复盘；
- 再次启动同一二级时仅轻提醒，不阻塞，也不自动带入旧项；
- 延迟补录先保存“当时结果”，不覆盖后来已变化的 WorkItem 当前状态；
- 离线可保存完成/取消意图，显示待同步；
- 离线不能新建正式三级 WorkItem，避免 PomodoroXII 成为第二任务事实源；
- 可直接对二级开始，或使用已缓存的现有三级项。

## 5. 执行化身模型

### 5.1 稳定语义

| 化身 | 主导执行模式 | 明确不表示 |
|---|---|---|
| 牛马 | 常规、稳定、按既定方式推进 | 低效或受罚 |
| 小猪 | 收窄注意力、埋头深挖、持续投入 | 必然高产 |
| 哈吉米 | 执行失序、方向混乱、受阻或恢复困难 | 失败人格 |
| 齐天大圣 | 跳出既定路径、产生创造性突破 | 最高等级或常态目标 |

### 5.2 规则

- 四种化身并列、互斥、不排名；
- 只对本轮实际触达的三级项选填；
- 未触达项不标注；
- 每个 `Session × 三级` 首版记录一个主导化身；
- 可补充“中途切换”和短说明；
- 用户主动标注，系统只可建议；
- 建议不得预选或自动提交；
- 化身与完成、有推进、卡住完全分离；
- 不将 Session 分钟按化身或三级项拆分；
- WorkItem 可派生显示最近化身和历史分布，但不能编辑成固定属性。

### 5.3 Session 总体层

- 有效性：决定是否计入投入；
- 总体推进：推进顺利 / 有推进 / 卡住 / 被打断；
- mood：个人情绪；
- 逐三级执行化身：具体成果的执行模式。

五组信息回答不同问题。有效性正常结束时默认成立；三级成果结果按本轮实际触达情况预填并允许修正；总体推进、Mood 与执行化身均可跳过。

## 6. 待处理区模型

### 6.1 三组问题

| 分组 | 来源 | 是否正式 WorkItem 状态 |
|---|---|---|
| 真正阻塞 | waiting、正式依赖未解除 | 是或由正式关系派生 |
| 推进异常 | 投入达到/超过上限、多轮未完成、过复盘点、反复哈吉米/卡住、主动关注 | 否，只是派生信号 |
| 待复盘 | Session 已结束但结果尚未确认 | 否，是 Session 流程状态 |

### 6.2 展示方式

- 当前 Space 有统一待处理入口；
- Project、二级卡和 L3 显示上下文轻提示；
- 推进异常以二级 WorkItem 为问题卡；
- 卡内列出三级原因和证据；
- 不将所有进行中 WorkItem 视为异常；
- 不使用统一红色“阻塞”标签覆盖三类问题。

### 6.3 处理动作

- 完成二级；
- 更新剩余估算；
- 标记范围扩大；
- 处理三级项：继续、暂停、等待、取消、重组；
- 延后到明确复盘点。

延后期间不重复打扰，到期重新出现。

## 7. 二级投入与成果复核

### 7.1 提醒真值

| 累计有效投入 | 提醒 |
|---|---|
| 低于预计下限 | 不提示 |
| 进入预计区间 | 只显示静态信息 |
| 达到或超过预计上限 | 显著成果复核卡 |

三级全部完成不改变提醒等级。用户始终可以主动提前完成二级。

### 7.2 显著复核卡

提供：

- 完成二级；
- 接受或修改剩余投入建议；
- 标记范围扩大；
- 标记或处理阻塞；
- 延后到复盘点。

复核卡不阻塞 Session 结束。用户直接关闭后，进入待处理区。

### 7.3 父子完成边界

- 二级完成时展示未完成三级清单；
- 系统不自动完成或取消三级；
- 仍需继续执行的三级必须移动到其他未完成二级，或重新打开原二级；
- 纯历史三级可以留在已完成二级的成果区。

## 8. 原型总结与组织设计

### 8.1 为什么不做一个超级原型

L3 画布、计时执行窗、结束复盘和待处理区分别解决不同认知任务。全部塞入单页会导致：

- 空间视觉与表单逻辑互相干扰；
- 无法判断问题来自领域语义还是 UI 密度；
- 每次修改一个专题都会破坏其他路径；
- 审查者容易把未冻结视觉当作领域事实。

因此采用：**一份领域总纲 + 四个专题原型 + 一个统一审查入口 + 历史实验归档。**

### 8.2 统一审查入口

建议新建 `prototype-review-hub.html`，只负责：

- 展示当前权威领域摘要；
- 选择审查角色与场景；
- 跳转四个专题原型；
- 展示每个原型回答的问题、已冻结语义和非目标；
- 标记旧原型为历史探索；
- 汇总审查意见，不承载完整业务交互。

### 8.3 四个专题原型

#### 原型 A：任务拆解与 Note

建议文件：`prototype-a-l3-decomposition.html`

验证：

- 二级展开/折叠三级成果；
- 创建、移动、提升三级；
- description 与 WorkItemNote 分工；
- Note 五种 Block 与两层列表；
- Checklist 与正式成果勾选的视觉区别；
- 已完成/取消三级进入成果与历史；
- 不验证最终动画质感和 WebGL 性能。

#### 原型 B：计时中 Orbit 执行窗

建议文件：`prototype-b-orbit-execution.html`

验证：

- 独立番茄钟启动后，Orbit 如何变身；
- 本轮成果清单与当前项行动 Checklist；
- 当前项切换；
- 打亮钩草稿与撤销；
- 从本轮移除与正式取消的区分；
- 执行化身标记与系统建议；
- 执行中新增三级的在线/离线边界。

#### 原型 C：Session 结束复盘

建议文件：`prototype-c-session-review.html`

验证：

- 有效性、三级成果结果、总体推进、Mood、逐项化身五组信息；
- 完成草稿统一提交；
- 未完成项默认保留、不自动结转；
- 延迟复盘与状态冲突；
- 二级达到预计上限后的显著复核卡；
- 复核卡不阻塞结束。

#### 原型 D：待处理与复核中心

建议文件：`prototype-d-attention-inbox.html`

验证：

- 真正阻塞、推进异常、待复盘三组；
- 二级问题卡与三级证据；
- 完成、重估、范围扩大、处理三级、延后复盘点；
- Project 与 L3 上下文轻提示；
- 当前 Space 内隔离与测试空间不聚合。

### 8.4 现有原型的归档定位

| 原型 | 保留价值 | 已失效语义 | 处理 |
|---|---|---|---|
| v1 `session-l3-hybrid-workbench.html` | 验证开始前/中/后创建三级的混合入口 | 三级承接计时、三级要求估时、结束结果模型过时 | 历史交互探索 |
| v2 `l3-multicenter-lifeline-review.html` | 验证 L2/L3 分层、多中心和 Note 分块 | 生命线编排和 UI 密度不是当前方向 | 历史布局探索 |
| v3 `l3-mindmap-canvas-review.html` | 验证“像思维导图一样拆解”理念 | 未覆盖二级计时归属、执行窗、化身和待处理区 | 视觉理念参考，不是业务总原型 |

旧文件不删除、不覆盖，但从“当前推荐审查”降级为历史档案。

## 9. 建议审查场景

### 场景 1：知道具体成果

从三级启动 → 自动归属二级 → Orbit 执行 → 打亮钩 → 结束提交 → 二级投入累计。

### 场景 2：暂时不知道怎么拆

从二级空计划启动 → 执行中发现三级 → 创建并设为当前项 → 添加 Note Checklist → 结束复盘。

### 场景 3：一轮处理多个三级

同一二级下选三个三级 → 依次切换当前项 → 两项触达、一项未触达 → 分别标记化身 → 不拆分钟。

### 场景 4：达到预计上限

二级累计投入达到上限 → 结束页显著复核 → 选择重估或延后 → 进入待处理区。

### 场景 5：阻塞与复盘积压

一个三级正式 waiting，一个三级多轮哈吉米，一个旧 Session 待复盘 → 三者进入不同分组，不共用状态。

### 场景 6：离线

离线选择缓存三级开始 → 打亮钩待同步 → 禁止新建正式三级 → 联网后幂等提交。

### 场景 7：单人多 Space

日常 Space 与测试 Space 拥有独立 Project、Session 和执行化身趋势，不跨空间汇总。

## 10. 已识别的规格冲突

现有正式或半正式规格尚未同步本轮决策，不能直接作为实施输入：

1. `WORKITEM_SHARED_CONTRACT.md` 仍含 `Workspace`、`assigneeId` 和多人事件字段。
2. `WORKITEM_RELATIONS_REVIEW.md` 仍使用递归完成百分比，当前三级只允许计数与状态分布。
3. `CYCLE_CAPACITY_REVIEW.md` 仍以成员自然周预算、成员分配和未负责人工作为核心。
4. Cycle 基线仍记录负责人和 `capacityMode`；当前 L3 模型改为二级唯一容量口径。
5. `REVIEW_GUIDE.md` 仍称三级是推荐计时归属，并要求三级预计投入。
6. v1 原型结束状态与当前 Session 五组结束信息模型不一致。
7. 旧规格允许一般 WorkItem 独立加入 Cycle，但当前需明确一级、二级、三级在单人产品中的 Cycle 资格。
8. 旧 `Workspace` 类型库作用域应重映射为当前 Pomodoro Space，而不是 Project Group。

这些冲突必须先通过规格修订解决，不能由工程实现自行猜测。

## 11. Non-goals

本轮总结与下一版原型明确不做：

- 最终 L2/L3 WebGL 视觉、轨道物理和动画参数；
- 多人协作、成员、邀请、负责人或权限；
- 跨 Pomodoro Space 任务、Session 或化身聚合；
- 三级 Session 分钟分配；
- 自动根据化身改变 WorkItem 状态；
- 自动根据累计时长完成二级；
- AI 自动创建、完成、取消 WorkItem；
- 化身排行榜、效率得分或人格评价；
- Note Checklist 参与任务进度；
- 一个页面覆盖全部产品能力。

## 12. 原型成功标准

下一版原型通过审查，至少满足：

1. 用户能区分二级投入、三级成果、Note 行动指导和 Session 时间。
2. 用户不会把两种勾选误认为同一种完成动作。
3. 用户能在无三级计划时立即开始专注。
4. 用户能在计时中切换当前项，但不会误以为系统在拆分分钟。
5. 用户能理解执行化身不是负责人、状态或效率等级。
6. 用户能区分真正阻塞、推进异常和待复盘。
7. 达到预计上限的复核不会阻止计时结束。
8. 日常 Space 与隐私/测试 Space 的业务数据和趋势不会串联。
9. 所有正式状态变化均能解释事实源与提交时机。
10. 审查者能够明确指出问题属于领域模型、流程还是 UI，而不是混在一起。

## 13. 下一步建议

在制作新原型前，按以下顺序推进：

1. 产品负责人审查并批准本总纲；
2. 修订 WorkItem 共享契约，撤销多人字段并加入 WorkItemNote；
3. 修订父子关系规格，删除递归百分比并加入二级/三级默认行为；
4. 修订 Cycle 容量规格为单人、二级容量口径；
5. 编写 FocusSession 集成契约、Outcome 与幂等命令；
6. 编写待处理派生规则；
7. 再分别制作四个专题原型；
8. 通过专题审查后，才制作统一演示壳或端到端高保真原型。

当前不建议直接在 v3 思维画布上继续叠加执行窗、化身和待处理区，否则会再次把视觉试验与领域事实绑死。
