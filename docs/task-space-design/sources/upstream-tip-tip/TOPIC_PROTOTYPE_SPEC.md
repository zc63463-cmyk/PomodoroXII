# TipTip × PomodoroXII 四专题原型规格 v1.0

> 状态：页面设计与验收基线
>
> 前置契约：`WORKITEM_SINGLE_USER_V11.md`、`CYCLE_CAPACITY_SINGLE_USER_V10.md`、`SESSION_TASK_INTEGRATION_V10.md`

## 1. 原型套件目标

四个原型分别验证一个高风险产品问题，不制作端到端超级原型：

| 原型 | 核心问题 | 主要对象 |
|---|---|---|
| A L3 拆解 | 用户能否区分二级范围、三级成果和 Note 行动指导？ | WorkItem / WorkItemNote |
| B Orbit 执行窗 | 用户能否在计时中低干扰地处理多个三级成果？ | SessionPlan / 当前项 / 化身 |
| C Session 复盘 | 五组结束信息和部分成功能否低负担共存？ | SessionOutcome / 命令状态 |
| D 待处理中心 | 用户能否区分阻塞、推进异常、待成果复盘与同步冲突？ | AttentionProjection |

统一 Hub 只负责导航、规则摘要和场景选择，不复制业务页面。

---

## 2. 统一视觉与交互约束

- 纯 2D、朴素高级感；浅色背景、低对比边界、克制色彩；
- 正式事实、Session 草稿、派生提醒使用不同语义色；
- 不使用多人头像、负责人、团队容量；
- 当前 Space 始终可见；不得出现跨 Space 聚合；
- 所有原型提供“设计规则”侧栏，可查看该操作修改哪类事实；
- 原型数据固定为单人场景，不接真实后端；
- 动画服务于状态变化，不用于装饰；支持减少动态效果。

---

# 原型 A：L3 任务拆解与 WorkItemNote

## A1. 页面目标

验证“像思维导图一样发散拆解你的任务”是否在不制造第四级任务的前提下成立。

## A2. 页面结构

```text
顶部：Space / Project / 一级范围 / 返回 L2
左侧：二级 WorkItem 列表
主画布：当前二级 + 三级成果节点 + Note 摘要
右侧：选中项详情与规则解释
底部：成果与历史折叠区
```

## A3. 必须交互

1. 单个二级展开/折叠三级；
2. 连续输入标题批量创建三级；
3. 三级创建时默认不展示估时和 Cycle 角色；
4. 点击三级查看 description 与 WorkItemNote 的区别；
5. Note 展示 paragraph、heading、ordered、unordered、checklist；
6. Checklist 可勾选，但页面明确“不影响成果状态”；
7. 列表项提升为 WorkItem 后，原 Note 项转为引用；
8. 三级继续拆解时显示“双出口”：添加 Note / 提升后拆解；
9. 已完成、已取消三级进入成果与历史；
10. 快速专注只携带上下文打开番茄钟，不在画布内运行完整计时器。

## A4. 状态场景

- 无三级的空状态；
- 有未开始/进行中/等待三级；
- Note 为空；
- Checklist 已部分勾选；
- 三级被依赖阻塞；
- 二级已完成但仍有活动三级的冲突提示。

## A5. 验收

- 用户不会把 Note Checklist 当作正式成果；
- 用户能说明为什么三级不独立计入 Cycle；
- 用户能在不填估时、类型或 Note 的情况下创建三级；
- 用户理解三级全部完成不等于二级自动完成。

## A6. Non-goals

- 最终 WebGL/动画质感；
- Session 运行态；
- Cycle 规划；
- 待处理中心。

# 原型 B：Orbit 计时中执行窗

## B1. 页面目标

验证一个 Session 归属一个二级、可处理多个同父三级且不拆分钟的执行体验。

## B2. 页面结构

```text
悬浮窗头部：计时 / 二级归属 / 有效状态
成果清单：本轮三级 WorkItem
当前项区：唯一当前三级
行动区：当前三级 WorkItemNote Checklist
化身区：牛马 / 小猪 / 哈吉米 / 齐天大圣
底部：随记 / 新增三级 / 结束
```

## B3. 必须交互

1. 切换唯一当前三级，不产生分钟分配；
2. 打亮钩形成可撤销完成草稿；
3. 从本轮移除，不改变 WorkItem；
4. “本项不再需要”二次确认并形成取消草稿；
5. 运行中新增已有三级或在线新建三级；
6. 离线状态下禁用正式新建并解释原因；
7. 当前项切换后显示对应 Note Checklist；
8. Checklist 勾选与成果钩视觉不同；
9. 用户主动选择化身，系统建议不预选；
10. 化身选填，离开已触达项时只轻提醒；
11. Session 随记可选片段沉淀到 Note，但不自动追加；
12. 结构发生变化时显示“本轮按启动快照继续”。

## B4. 状态场景

- 无三级计划直接开始；
- 三个成果项、一个当前项；
- 一个完成草稿、一个卡住、一个未触达；
- 离线；
- 任务被移动/取消后的冲突；
- 系统建议哈吉米但用户选择牛马。

## B5. 验收

- 用户知道当前项不是分钟归属；
- 用户能撤销完成草稿；
- 用户不会把 Note Checklist 的勾选理解为完成任务；
- 用户知道化身是执行模式，不是负责人或效率等级。

## B6. Non-goals

- 完整 Session 结束复盘；
- 自动识别化身；
- 多任务分钟时间轴；
- 多人协作。

# 原型 C：Session 结束复盘

## C1. 页面目标

验证时间事实、成果命令、自我标定和同步状态能否以渐进披露方式完成。

## C2. 页面结构

```text
第一屏：时间有效性 + 三级成果结果
第二层折叠：总体推进 / Mood / 逐项化身
提交结果：逐项命令回执
条件卡：二级达到预计上限时显著复核
```

## C3. 必须交互

1. 正常结束默认计入；提前结束选择保留投入/误启动；
2. 未触达项预填 untouched，支持批量确认；
3. 完成草稿确认后提交正式完成命令；
4. 未完成项默认原地保留，可选下轮候选/暂停/等待/取消；
5. 整组成果可稍后复盘或显式跳过；
6. 跳过不生成任务命令；
7. 总体推进、Mood、逐项化身可跳过；
8. 部分成功：3 项成功、1 项失败/冲突/未知；
9. 单项重试使用原 commandId；
10. 离线显示“结果已保存，任务状态待同步”；
11. 延迟复盘时展示当时快照与当前状态差异；
12. 达到二级预计上限显示显著复核卡，但允许延后且不阻止结束；
13. 提供可展开“复核二级成果”入口；
14. 完成二级时处理仍活动的三级。

## C4. 状态场景

- 正常完成、全部成功；
- 提前结束但投入有效；
- 没有三级计划；
- 延迟复盘；
- 在线部分成功；
- 离线待同步；
- 二级达到预计上限；
- 二级完成但有活动三级。

## C5. 验收

- 用户能先保存时间，不被复盘阻塞；
- 用户能区分“待成果复盘”和“待同步”；
- 用户理解化身、Mood、总体推进均不是任务状态；
- 单项失败不会让已成功项回滚；
- 三级全完成不会自动弹出二级完成判决。

## C6. Non-goals

- L3 画布；
- 长期统计仪表盘；
- 命令实现细节；
- 自动重估。

# 原型 D：待处理与复核中心

## D1. 页面目标

验证长期问题治理是否清楚且不会形成“红点坟场”。

## D2. 页面结构

```text
顶部：当前 Space / 分组计数 / 解释
分组一：真正阻塞
分组二：推进异常
分组三：待成果复盘
分组四：待同步与冲突
右侧：选中二级问题卡详情
```

## D3. 必须交互

1. 真正阻塞展示正式状态或依赖证据；
2. 推进异常以二级问题卡为主，列出三级原因；
3. 展示实际投入 / 预计区间；
4. 处理动作：完成二级、重估、范围扩大、处理三级原因；
5. 暂不处理必须设置复盘点；
6. 待成果复盘打开原 Session；
7. 待同步与冲突展示命令级回执和重试；
8. Project、二级卡、L3 的提示跳转到同一条派生问题；
9. Space 切换后数据完全隔离；
10. 空状态强调“当前没有需要处理的问题”，不制造焦虑。

## D4. 状态场景

- 外部依赖未解除；
- 二级达到预计上限；
- 三级连续三轮未完成且两次哈吉米；
- Session 成果待复盘；
- 一个失败命令、一个版本冲突；
- 已设置未来复盘点，当前静默；
- 测试 Space 与日常 Space 数据不同。

## D5. 验收

- 用户能准确区分四个分组；
- 用户不会把推进异常当成正式阻塞状态；
- 延后后问题不会反复打扰；
- 同一问题不会在多个入口形成可写副本；
- 测试/隐私 Space 不污染当前统计。

## D6. Non-goals

- 跨 Space 总览；
- 绩效评分；
- 自动修复任务；
- 团队负责人队列。

---

# 3. 统一审查 Hub

## Hub 内容

- 当前三份权威契约；
- 四个专题原型入口；
- 每个专题核心问题、场景和 Non-goals；
- 历史 v1/v2/v3 原型归档；
- 审查意见记录区；
- 当前批准状态。

## Hub 验收

- 用户能在 30 秒内知道先看哪个原型；
- 历史原型明确标注“探索档案”；
- Hub 不承载具体业务编辑；
- 所有专题使用相同术语和示例数据。

# 4. 建议制作顺序

1. A：先固定实体视觉边界；
2. B：再验证计时中操作；
3. C：验证提交与复盘负担；
4. D：验证长期治理；
5. Hub：最后汇总入口和批准状态。
