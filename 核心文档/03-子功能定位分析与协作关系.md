## 二、子功能定位分析（精确到子功能级别）

### A. 番茄钟核心

#### A1. 计时器引擎

| 维度 | 分析 |
|------|------|
| **输入** | 用户点击开始/暂停/停止；配置参数（工作时长、休息时长、长休息间隔） |
| **输出** | 倒计时 UI；会话记录（started_at, ended_at, duration, type, completed） |
| **依赖** | useWakeLock（防止屏幕熄灭）、useAudio（完成提示音）、session store (Zustand) |
| **边界** | 不负责会话事件追踪（A2）、不负责统计（F1） |
| **成熟参考** | **Forest App**（专注计时 + 游戏化）、**Toggl Track**（时间追踪 + 报告）、**Pomofocus**（极简番茄钟） |
| **PomodoroXII 改进** | 加 HLC 时间戳解决跨设备时钟偏差；会话 duration 用服务端验证 |

#### A2. 会话事件追踪

| 维度 | 分析 |
|------|------|
| **输入** | 计时器状态变化（暂停/恢复/中断/分心检测） |
| **输出** | session_events 表（event_type, timestamp, metadata） |
| **依赖** | A1 计时器引擎 |
| **边界** | 不负责认知评估（A4），只记录原始事件 |
| **成熟参考** | **Toggl Track**（时间条目 + 标签）、**RescueTime**（活动追踪 + 分类） |
| **PomodoroXII 改进** | 事件流导出为 NDJSON 供 Agent 分析 |

#### A3. 会话上下文

| 维度 | 分析 |
|------|------|
| **输入** | 环境传感器（如果可用）、用户手动输入、设备信息 |
| **输出** | session_contexts 表（location, weather, device, environment） |
| **依赖** | A2 会话事件 |
| **边界** | 隐私敏感，导出时需 anonymized 级别处理 |
| **成熟参考** | **RescueTime**（自动环境检测）、**Freedom**（干扰阻断 + 上下文） |

#### A4. 认知标记

| 维度 | 分析 |
|------|------|
| **输入** | 用户自评（注意力 1-10、心流 yes/no）、A2 事件推断 |
| **输出** | cognitive_marks 表（attention_score, flow_state, golden_hour） |
| **依赖** | A2 会话事件、F2 专注力模式（反馈循环） |
| **边界** | 主观数据，不做医疗级判断 |
| **成熟参考** | **Rise Science**（睡眠周期 + 黄金时段）、**Centered**（心流状态追踪） |

### B. 任务管理

#### B1-B5. 任务 CRUD + 状态机 + 关联 + 标签 + 优先级

| 维度 | 分析 |
|------|------|
| **输入** | 用户创建/编辑任务（title, description, priority, tags, due_date） |
| **输出** | tasks 表 + sync 事件 |
| **依赖** | B3 依赖 A1（任务关联会话）、G1 Outbox（变更入队） |
| **边界** | 不负责会话计时（A1），不负责统计（F1） |
| **成熟参考** | **Todoist**（任务管理标杆，Karma 系统）、**TickTick**（番茄钟 + 任务集成）、**Things 3**（GTD 方法论） |
| **PomodoroXII 改进** | tags 导出为原生 list；actual_pomodoros 服务端聚合（已有）；加 version 字段 |

### C. 笔记系统

#### C1. 笔记 CRUD（双存储）

| 维度 | 分析 |
|------|------|
| **输入** | 用户创建/编辑笔记（title, content, tags, folder_id） |
| **输出** | pomodoroxi.db notes 表（元数据）+ .md 文件（正文）+ index.db（索引） |
| **依赖** | C2 文件夹树、G5 墓碑（删除同步） |
| **边界** | 正文只存 .md（唯一 SoT），DB 只存元数据 + content_hash |
| **成熟参考** | **Obsidian**（纯文件 + frontmatter）、**Joplin**（DB 索引 + .md 文件 + REST API）、**Notion**（块级编辑 + API） |
| **PomodoroXII 改进** | .md 带 YAML frontmatter；content_hash 双游标增量 |

#### C2. 文件夹树

| 维度 | 分析 |
|------|------|
| **输入** | 用户创建/移动/删除文件夹 |
| **输出** | folders 表（parent_id 自引用，UniqueConstraint 同级不重名） |
| **依赖** | C1 笔记（folder_id FK）、级联删除（CascadeService BFS） |
| **成熟参考** | **Obsidian**（无限嵌套文件夹）、**Joplin**（笔记本 + 子笔记本） |

#### C3. 全文搜索

| 维度 | 分析 |
|------|------|
| **输入** | 搜索查询（中文子串 ≥3 字符，短查询 LIKE 回退） |
| **输出** | note_id + title + excerpt + score |
| **依赖** | C1 笔记（FTS5 索引基于 .md 正文） |
| **边界** | 只搜索笔记，不搜索任务/复盘 |
| **成熟参考** | **Obsidian**（客户端 Omnisearch 插件）、**Logseq**（内置全文搜索）、**Notion**（服务端搜索） |
| **PomodoroXII 优势** | FTS5 trigram 服务端搜索，比 Obsidian 客户端索引更轻量 |

#### C4. 版本历史

| 维度 | 分析 |
|------|------|
| **输入** | edit_note 触发（SHA-256 比对，内容变化才备份） |
| **输出** | .meta/version_backups/{version_id}.md + note_versions 表 |
| **依赖** | C1 笔记 |
| **成熟参考** | **Obsidian Sync**（远程全版本，1-12 月保留）、**Notion**（页面历史）、**Git**（行级 diff） |
| **PomodoroXII 优势** | 本地化版本备份，不依赖远程服务 |

### D. 速记系统

#### D1-D7. 速记 CRUD + 心情 + 关联 + 评论 + 标签树

| 维度 | 分析 |
|------|------|
| **输入** | 快速记录（content, mood, tags, pinned）+ 可选关联（session/task/schedule） |
| **输出** | quick_notes 表 + 关联表（session_quick_notes, task_quick_notes, schedule_quick_notes） |
| **依赖** | A1 会话（D3）、B1 任务（D4）、C6 转换（速记→笔记） |
| **边界** | 速记是轻量记录，笔记是结构化内容，两者通过 convert 桥接 |
| **成熟参考** | **Flomo**（浮墨笔记，卡片笔记 + 标签）、**Google Keep**（速记 + 颜色标签）、**Apple Notes**（速记 + 文件夹） |
| **PomodoroXII 优势** | 速记天然关联番茄钟会话，可追溯"这个想法是在哪个专注时段产生的" |

### E. 复盘系统

#### E1-E6. 每日复盘 + 结构化 + 模板 + 提示 + 标签 + 自动关联

| 维度 | 分析 |
|------|------|
| **输入** | 用户每日填写（mood, energy, achievements, challenges, learnings） |
| **输出** | reflections 表 + reflection_templates |
| **依赖** | A1 会话数据（自动填充统计）、E6 自动关联（AI 匹配相关复盘） |
| **边界** | 不负责统计分析（F1），只负责记录 |
| **成熟参考** | **Day One**（日记 + 模板 + 天气）、**Stoic**（哲学日记 + 提示）、**Reflectly**（AI 辅助日记） |
| **PomodoroXII 优势** | 复盘天然关联番茄钟数据，自动填充"今天专注了多久" |

### F. 统计分析

#### F1-F7. 统计 + 模式 + 洞察 + 日历 + 报告 + 导出

| 维度 | 分析 |
|------|------|
| **输入** | sessions/tasks/reflections 数据 |
| **输出** | 聚合统计（JSON）、可视化图表、导出文件（MD/CSV/JSON/NDJSON/Arrow） |
| **依赖** | A1 会话（F1/F2）、A4 认知（F2/F3）、E1 复盘（F5）、全部数据（F7） |
| **边界** | F7 导出是跨功能域的，其他统计各自聚焦 |
| **成熟参考** | **Toggl Track**（时间追踪 + 报告 + 导出）、**RescueTime**（生产力评分 + 周报）、**Clockify**（时间追踪 + API 导出） |
| **PomodoroXII 改进** | 后端 Export API（复用 export-v2.ts 的 _graph/_analysis_hints）+ MCP Server |

### G. 同步引擎

#### G1-G6. Outbox + 云端 + 门面 + 冲突 + 墓碑 + 事务

| 维度 | 分析 |
|------|------|
| **输入** | 客户端变更（Outbox 事件）、服务端变更（pull 增量） |
| **输出** | DB 写入 + .md 文件写入 + tombstone + sync_events 审计 |
| **依赖** | 全部业务实体（G4 冲突检测需要 updated_at + content_hash） |
| **边界** | 不负责业务逻辑，只负责数据传输和冲突解决 |
| **成熟参考** | **PowerSync**（SQLite 同步标杆）、**Replicache**（Mutator 模式）、**Obsidian Sync**（文件级 + diff-match-patch） |
| **PomodoroXII 改进** | sync_safety.py 提取 5 道安全检查；Saga 模式跨库一致性；content_hash 双游标 |

## 三、功能协作关系图

### 3.1 核心数据流

```
用户操作
    │
    ├──→ [A1 计时器] ──→ [A2 事件追踪] ──→ [A4 认知标记]
    │         │                                  │
    │         ├──→ [B3 任务关联] ←── [B1 任务CRUD]
    │         │                        ├──→ [B4 标签]
    │         │                        └──→ [B5 优先级]
    │         │
    │         ├──→ [D3 速记关联] ←── [D1 速记CRUD]
    │         │         │              ├──→ [D2 心情]
    │         │         │              ├──→ [D6 评论]
    │         │         │              └──→ [D7 标签树]
    │         │         │
    │         │         └──→ [C6 转换] ──→ [C1 笔记CRUD] ──→ [C3 FTS5搜索]
    │         │                           ├──→ [C4 版本历史]
    │         │                           └──→ [C5 回收站]
    │         │                           └──→ [C2 文件夹树]
    │         │
    │         ├──→ [I1 日程] ──→ [I2 日历视图]
    │         │         └──→ [I3 日程-速记关联]
    │         │
    │         ├──→ [J1 习惯] ──→ [J2 打卡] ──→ [J3 连续天数]
    │         │
    │         ├──→ [K1 时间块] ──→ [K2 类型] ──→ [K3 状态]
    │         │
    │         └──→ [E1 复盘] ←── [E6 自动关联]
    │                       ├──→ [E2 结构化]
    │                       ├──→ [E3 模板]
    │                       └──→ [E4 提示]
```

> **前端实现方式变化说明**：上述数据流与业务逻辑不变，仅前端实现从 Vue 迁移至 React 19，映射关系如下：
>
> | 维度 | pomodoroxi (Vue 3) | PomodoroXII (React 19) |
> |------|---------------------|------------------------|
> | **组件** | `.vue` 单文件组件（`<template>`） | `.tsx` 组件（JSX） |
> | **路由** | vue-router 4.4（编程式路由） | Next.js App Router（文件系统路由） |
> | **复用逻辑** | 35+ 个 composables（`useXxx.ts`） | 35+ 个 React hooks（`useXxx.ts`，命名保持 `use` 前缀） |
> | **响应式 API** | `ref()` / `reactive()` / `computed()` | `useState()` / `useMemo()` / Zustand selector |
> | **副作用** | `watch()` / `watchEffect()` | `useEffect()` |
> | **客户端状态** | 17 个 Pinia store（`defineStore`） | 17 个 Zustand store（`create()`），`storeToRefs` → selector 解构 |
> | **服务端缓存** | （无，sync 直连） | TanStack Query v6（`useQuery` / `useMutation`，缓存 + 乐观更新 + 去重） |
> | **表单** | `v-model` 双向绑定 | React Hook Form + Zod（非受控 + 运行时类型验证） |
> | **组件通信** | `defineProps()` / `defineEmits()` | props / callback props |
> | **依赖注入** | `provide` / `inject` | React Context |
> | **测试** | `@vue/test-utils` | `@testing-library/react` |
> | **PWA** | vite-plugin-pwa | @serwist/next |

### 3.2 耦合点分析

| 耦合点 | 涉及功能 | 耦合类型 | 风险 | 建议 |
|--------|---------|---------|------|------|
| task_id in sessions | A1↔B3 | 数据 FK | 低 | 保留 |
| session_id in quick_notes | A1↔D3 | 数据 FK | 低 | 保留 |
| folder_id in notes | C1↔C2 | 数据 FK | 中 | 级联删除需 CascadeService |
| C6 转换（速记→笔记） | D1→C1 | 跨域写 | 中 | Saga 模式 |
| habit_id in habit_check_ins | J1↔J2 | 数据 FK + 级联 | 中 | CascadeService 统一处理 |
| schedule_id in schedule_quick_notes | I1↔D5 | 数据 FK + 级联 | 中 | CascadeService 统一处理 |
| sync 双路径分叉 | G4↔C1 vs G4↔其他 | 架构分叉 | 高 | note/folder 走 NoteFileService，其他走 ORM |
| F4 活动日历多源 | A1+J2→F4 | 多源聚合 | 低 | 日历展示会话+习惯打卡 |
| F7 导出 | 全部→F7 | 读依赖 | 低 | 后端 Export API |
| G4 冲突检测 | 全部→G4 | 读 updated_at | 中 | + content_hash |
| G5 墓碑 | 全部→G5 | 写依赖 | 高 | TombstoneService 统一 |
| G6 事务 | 全部→G6 | 写依赖 | 高 | useTransaction 修复表列表 |
| Zustand store 间依赖（selector 嵌套） | 跨域 store（如 I1↔D5、J2↔F4） | 状态依赖 | 中 | 跨 store 调用通过 `get()` 显式注入；使用 `shallow` 比较避免循环依赖与多余 re-render；避免在 selector 中直接订阅其他 store 导致渲染级联 |
| TanStack Query cache 与 Dexie 同步策略 | F/G ↔ Dexie 离线存储 | 缓存一致性 | 中 | `useQuery` 优先读 Dexie 离线缓存，`useMutation` 乐观更新后 `invalidateQueries`；`gcTime` 与 Dexie 保留期对齐；Outbox 队列 (`_dirty`) 作为 cache miss 兜底 |
