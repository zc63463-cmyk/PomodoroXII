## 一、背景：为什么需要这份文档

2026 年 Vibe Coding 已被 92% 开发者使用，但代码重构比例从 25% 降到不足 10%，AI 生成的代码重大问题是人类代码的 1.7 倍。核心矛盾是：**AI 实现能力越来越强，但项目理解能力跟不上**。

这份文档的目标不是教 AI 怎么写代码，而是**告诉 AI 这个项目的约束、不变量和操作边界**——让下一版更强的 AI 工具能安全地改进功能而不破坏已有行为。

参考的行业标准：

| 标准 | 来源 | 用途 |
|------|------|------|
| **AGENTS.md** | OpenAI 2025.08 推出，Linux 基金会 Agentic AI Foundation 托管 | 跨工具通用 Agent 指令（Codex/Copilot/Cursor/Jules/Aider） |
| **CLAUDE.md** | Anthropic Claude Code 官方 | Claude Code 专属项目规则 |
| **.claude/rules/** | Claude Code 路径匹配规则 | 按文件路径加载特定规则 |

## 二、AGENTS.md 规范要点

### 2.1 文件格式

- 纯 Markdown，无强制字段，无严格格式要求
- 放在项目根目录，AI 工具自动读取，无需额外配置
- 支持**嵌套**——monorepo 中每个子项目可以有独立的 AGENTS.md

### 2.2 推荐结构

```markdown
# [项目名] Agent Guide

## Dev environment tips
- 构建命令、开发环境配置、快速跳转方式

## Testing instructions
- 测试命令、CI 流程、合并前必须通过的检查

## Coding conventions
- 架构分层规则、命名约定、禁止事项

## Project-specific guidance
- 项目特有的坑点、不变量、架构决策理由

## PR instructions
- 提交格式、分支命名、审查流程
```

### 2.3 CLAUDE.md 与 AGENTS.md 的关系

| 维度 | CLAUDE.md | AGENTS.md |
|------|-----------|-----------|
| 谁读 | Claude Code 专属 | 跨工具（Codex/Cursor/Jules/Aider） |
| 定位 | 项目规则 | 通用 Agent 指令 |
| 复用 | 可 `@AGENTS.md` 导入 | 独立存在 |

**推荐策略**：项目根目录放 AGENTS.md（跨工具兼容），Claude Code 专属指令放 CLAUDE.md 并 `@AGENTS.md` 导入。

## 三、CLAUDE.md 最佳实践

### 3.1 该写什么（五类内容）

| 类别 | 说明 | 判断标准 |
|------|------|---------|
| **技术栈和版本** | 框架版本差异是 AI 犯错源头 | "如果删掉这行，AI 会不会用错版本？" |
| **常用命令** | 构建/测试/lint/启动 | "AI 需要知道怎么验证自己的改动吗？" |
| **架构决策和理由** | 不光写规则，写"为什么" | "AI 能从代码本身推断出这个理由吗？" |
| **团队约定和坑点** | 提交格式、环境依赖、已知问题 | "这是踩过坑才得到的教训吗？" |
| **当前任务** | 任务描述、验收标准、阻塞问题 | "AI 需要知道当前在做什么吗？" |

### 3.2 不该写什么

| 不该写 | 原因 |
|--------|------|
| 代码风格（缩进/import 排序） | 交给 ruff/prettier 等格式化工具 |
| 语言默认行为（"Python 用 f-string"） | AI 已知，写下来是噪音 |
| 大段 API 文档 | 放链接，AI 需要时自己读 |
| 正确废话（"写干净的代码"） | 对 AI 几乎无约束力 |

### 3.3 规则编写原则

1. **具体可验证**：❌ "注意代码可读性" → ✅ "函数名用动词开头，单个函数不超过 40 行"
2. **禁令配替代方案**：❌ "不要用字段注入" → ✅ "不要用字段注入；用构造器注入 + @RequiredArgsConstructor"
3. **善用标记词但不滥用**：IMPORTANT: 只用于反复违反的规则
4. **标题用常规名字**：Commands/Architecture/Conventions/Testing
5. **软约束 vs 硬约束**：CLAUDE.md 是软约束，Linter/Hook/CI 是硬约束。能用工具强制的不要写成自然语言

### 3.4 项目变大后的分层策略

| 阶段 | 策略 | 行数建议 |
|------|------|---------|
| 起步 | 单文件，全部规则在一个 CLAUDE.md | ≤200 行 |
| 拆分 | 主文件做路由，`@docs/xxx.md` 引用细节 | 主文件 ≤50 行 |
| 按区域加载 | `.claude/rules/` + frontmatter paths 匹配 | 按需加载 |

### 3.5 维护原则

- **添加要慢**：只在 AI 确实犯了错误且规则能防止同类错误时才写入
- **删除要果断**：删掉后 AI 行为没变化 → 规则从未起作用
- **定期审查**：问 AI "如果我删掉这条规则，你会改变行为吗？"

## 四、Vibe Coding 工程化最佳实践（2026）

### 4.1 五条工程化实践

| 实践 | 内容 |
|------|------|
| **分层接受标准** | 低风险（脚本）基本测试；中风险（业务逻辑）完整 review + 单测；高风险（认证/支付）安全审计 |
| **架构先行** | 先和 AI 讨论架构，批准后再生成代码。不要直接说"帮我做个XX" |
| **强制测试覆盖** | AI 生成代码必须配合测试覆盖率报告，没有测试不允许合入主分支 |
| **定期人类重构日** | 每两周专门拿出一天对 AI 代码重构，偿还 AI Debt |
| **工具链集成** | SAST + code review + 测试覆盖集成到 CI/CD，AI 代码和人类代码接受相同质量门禁 |

### 4.2 子系统可扩展性设计五原则

| 原则 | 核心思想 | PomodoroXII 对应 |
|------|---------|-----------------|
| **契约先行，实现可换** | 接口用 Protocol/ABC 定义，AI 改实现不改签名 | Service 层 + interfaces.py |
| **子系统自包含** | 每个子系统能独立放进 AI 上下文窗口 | file_system 15 文件自包含 |
| **测试即规格** | 测试是 AI 的验收标准，测试 = 不变量 | 每个 Service 方法 3+ 测试 |
| **功能隔离，新增不改旧** | 新功能走新路径，用 registry 模式做路由 | ENTITY_REGISTRY + sync 双路径 |
| **元数据自描述** | AI 能自动理解项目结构和约束 | AGENTS.md + frontmatter + CLAUDE.md |

### 5.2 根目录 AGENTS.md 模板

```markdown
# PomodoroXII Agent Guide

番茄钟 + 笔记 + 复盘 + 习惯的个人状态管理平台。
Python 3.13 / FastAPI / SQLAlchemy 2.0 / React 19 + Next.js 15 / Zustand / shadcn/ui / Dexie.js / SQLite

## Commands
- 后端启动: `cd backend && uv run uvicorn app.main:app --reload --loop uvloop`
- 后端测试: `cd backend && uv run pytest -v`
- 前端启动: `cd frontend && next dev`
- 前端测试: `cd frontend && npx vitest run`
- 前端构建: `cd frontend && next build`
- 数据库迁移: `cd backend && uv run alembic upgrade head`
- 代码检查: `cd backend && uv run ruff check . && cd ../frontend && npx eslint .`

## Architecture
- 后端三层架构: routes (校验+commit) → services (flush+业务逻辑) → models (纯数据)
- 前端框架: React 19 + Next.js 15 (App Router), Turbopack 构建
- 前端状态: Zustand 5 客户端状态 + TanStack Query v6 服务端状态
- 前端 UI: shadcn/ui + Radix UI + Tailwind CSS v4
- 前端表单: React Hook Form + Zod 运行时验证
- 前端离线: Dexie.js 4.4 + @serwist/next (PWA Service Worker)
- 双存储: pomodoroxi.db (元数据) + file_system (.md 正文 + index.db)
- 同步引擎: LWW + Tombstone + Outbox, note/folder 走 NoteFileService 桥接
- 11 功能域, 60 子功能, 14 种同步实体

## Must Follow
- Routers commit, Services 只 flush 不 commit
- Services 层不导入 FastAPI (业务逻辑与 HTTP 解耦)
- Models 层只有字段和约束, 无业务方法
- 新增实体注册到 ENTITY_REGISTRY, 不修改已有注册项
- 级联删除走 CascadeService, REST 和 sync 共用
- sync push 的 5 道安全检查必须复用 sync_safety.py
- .md 文件必须包含 YAML frontmatter
- 时间戳 DB 内无 Z 后缀, 导出/frontmatter 加 Z
- React Compiler 已启用 (next.config.ts), 无需手动 useMemo/useCallback
- Server Component 为默认, 需要交互/hooks 的组件在文件顶部声明 'use client'
- 客户端状态用 Zustand store, 服务端数据用 TanStack Query hook, 职责不混用
- 表单用 React Hook Form + Zod, 不用受控双向绑定模式
- 路由走 Next.js App Router 文件系统约定 (app/ 目录), 不用编程式路由

## Never Do
- 不要在 Service 层调用 db.commit()
- 不要在 sync adapter 中跳过安全检查
- 不要直接操作 .md 文件, 通过 file_system 接口
- 不要在 models 中写业务方法
- 不要用 python-jose, 用 PyJWT
- 不要用 passlib, 直接调用 bcrypt
- 不要硬编码路径, 用 settings 配置项
- 不要用 Vue (本项目已迁移到 React 19)
- 不要用 Redux (用 Zustand 做客户端状态管理)
- 不要混用 Server/Client Components (Server Component 不可直接导入 Client Component 内部模块; Client Component 可通过 props 接收 Server Component 传递的可序列化数据)
- 不要手动 useMemo/useCallback (React Compiler 已启用, 除非有性能测量依据)
- 不要用 v-model 语法 (React 无双向绑定, 用 React Hook Form)

## Testing
- 每个 Service 方法至少 3 个测试: 正常路径 + 边界条件 + 错误处理
- 测试名用中文描述行为: test_删除任务时级联删除关联会话并创建墓碑
- 测试用内存 SQLite, 不依赖外部服务
- 前端组件测试用 @testing-library/react, 以用户行为驱动断言
- AI 生成代码必须配合测试, 没有测试不允许合入

## Project-specific
- Sync 引擎是核心, 改动前必须读 01-深度架构规划.md
- file_system 是自包含子系统, 改动前必须读其 AGENTS.md
- content 字段已移除, .md 为唯一 source of truth
- content_hash 驱动增量同步, 不要用 updated_at 单游标
- 前端从 Vue 迁移到 React 19, 技术栈决策见 02-技术栈升级推荐.md
- 框架无关代码 (services/types/utils/workers) 可直接复用, 组件层需重写为 React
```

#### 5.3.1 file_system 模板（后端，框架无关）

```markdown
# file_system 子系统 Agent Guide

## What this is
从 video4KB 移植的 .md 文件管理系统。8 Mixin 组合, 三层锁, FTS5 trigram 搜索。
自包含, 仅依赖 filelock/nanoid/python-slugify 三个第三方包。

## Key files
- engine/__init__.py — 8 Mixin 组合为 FileSystemStorage
- engine/note_ops.py — 笔记 CRUD + 批量读取
- engine/search_ops.py — FTS5 搜索 + LIKE 回退
- schema.py — index.db 7 表 + FTS5 表结构
- frontmatter.py — .md YAML frontmatter 读写

## Interface
- FileSystemStorage(root_dir, index_db) — 构造函数
- create_note(title, content, folder_id, external_id) — 创建笔记
- read_note(note_id) — 读取正文
- read_notes_batch(note_ids) — 批量读取
- edit_note(note_id, content) — 编辑（自动版本备份）
- search(query, limit) — FTS5 搜索
- delete_note(note_id) — 软删除（移到 .trash/）

## Invariants (不可破坏)
- .md 文件必须包含 YAML frontmatter
- 写操作必须经过三层锁: RLock → FileLock → SQLite 事务
- 文件写入用临时文件 + os.replace 原子操作
- edit_note 时 SHA-256 比对, 内容变化才版本备份
- index.db 有 7 表 + 1 FTS5, 不要遗漏 schema_meta 和 sync_audit_log

## How to extend
- 新增搜索类型: 在 search_ops.py 加方法, 不改 note_ops
- 新增文件类型（如复盘）: 新建 reflection_ops.py Mixin, 加到 __init__.py 组合
- 不改 base.py 的锁/连接/路径逻辑, 除非你完全理解三层锁设计
```

```markdown
# frontend 子系统 Agent Guide

## What this is
React 19 + Next.js 15 (App Router) 前端应用。
Zustand 5 管理客户端状态, TanStack Query v6 管理服务端状态。
shadcn/ui + Radix UI + Tailwind CSS v4 构建 UI 组件。
Dexie.js 4.4 离线存储, @serwist/next 提供 PWA Service Worker。

## Key files
- src/app/layout.tsx — 根布局, 注入 Providers (QueryClient / Theme)
- src/app/providers.tsx — 全局 Provider 组合 (TanStack Query / Next-Theme)
- src/app/page.tsx — 首页 (Server Component, 服务端获取数据)
- src/stores/*.ts — Zustand stores (客户端状态, 一个领域一个 store)
- src/lib/query-client.ts — TanStack QueryClient 配置 (staleTime / retry 策略)
- src/lib/validations/*.ts — Zod schema 定义 (表单验证 + 运行时类型校验)
- src/services/api.ts — Axios 客户端 + Cloudflare 重试逻辑 (框架无关, 从 pomodoroxi 迁移)
- src/services/database.ts — Dexie.js v15 schema (框架无关, 从 pomodoroxi 迁移)
- src/components/ui/* — shadcn/ui 组件源码 (CLI 生成, 完全可控)
- next.config.ts — Next.js 配置 (React Compiler / Serwist PWA / 路径别名)

## Interface
- app/[route]/page.tsx — 页面组件, 文件系统路由 (替代 vue-router)
- app/[route]/[id]/page.tsx — 动态路由, params.id 获取路径参数
- app/api/[route]/route.ts — Route Handlers (BFF 层, 如需服务端代理)
- useXxxStore(selector) — Zustand store 调用, 必须用 selector 避免不必要 re-render
- useQuery({ queryKey, queryFn }) — TanStack Query 数据获取 hook
- useMutation({ mutationFn, onMutate, onError }) — TanStack Query 变更 hook (乐观更新)
- useForm({ resolver: zodResolver(schema) }) — React Hook Form 表单

## Invariants (不可破坏)
- Server Component (默认) 不能使用 useState/useEffect/handlers, 只做数据获取和静态渲染
- Client Component 必须在文件首行声明 'use client', 不能用 async export default
- Client Component 不能直接导入 Server Component, 只能通过 children prop 传递
- Zustand store 必须用 selector 访问状态: useStore(s => s.field), 不要解构整个 store
- TanStack Query mutation 必须实现乐观更新: onMutate 写 Dexie + 更新缓存, onError 回滚
- shadcn/ui 组件在 components/ui/ 下, 通过 npx shadcn add 添加, 不手动创建
- 路由由文件系统约定, 不用编程式路由注册
- React Compiler 已启用, 禁止无依据的手动 useMemo/useCallback

## How to extend
- 新增页面: 在 src/app/ 下新建目录 + page.tsx, 零配置路由
- 新增客户端交互组件: 新建 .tsx 文件, 首行加 'use client', 放 src/components/
- 新增 UI 组件: 运行 npx shadcn@latest add [component], 不手动写 components/ui/
- 新增状态领域: 在 src/stores/ 下新建 [domain]-store.ts, 用 create()(devtools(persist(...)))
- 新增数据获取: 在 src/lib/hooks/ 下新建 use-xxx.ts, 封装 useQuery + Dexie 回退逻辑
- 新增表单: 在 src/lib/validations/ 下新建 zod schema, 组件用 useForm + zodResolver
- 新增 Route Handler: 在 src/app/api/ 下新建 route.ts, 导出 GET/POST 等 HTTP 方法
```

## 六、AI 编码工作流指导

### 6.1 改进已有功能的流程

```
1. 读 AGENTS.md → 理解项目约束
2. 读子系统 AGENTS.md → 理解子系统边界
3. 读对应 Service 的测试 → 理解不变量
4. 改实现 → 保证接口签名不变
5. 跑测试 → 全绿才能提交
6. 更新测试 → 如果行为有变化, 更新测试描述
```

### 6.2 扩展新功能的流程

```
1. 读 AGENTS.md → 理解项目架构
2. 读 ENTITY_REGISTRY → 理解注册模式
3. 新建 Service 文件 → 不修改已有 Service
4. 新建路由文件 → 注册到 v1 __init__.py
5. 新建测试文件 → 至少 3 个测试
6. 如果需要同步 → 注册到 ENTITY_REGISTRY + sync adapter
7. 如果需要文件化 → 新建 file_system Mixin, 不改已有 Mixin
8. 前端新建页面 → 在 src/app/ 下新建目录 + page.tsx (Next.js 文件系统路由), 不用 Vue SFC
9. 前端交互组件 → 新建 'use client' 组件, 放 src/components/
10. 前端状态 → Zustand store (客户端状态) 或 TanStack Query hook (服务端数据)
11. 前端表单 → Zod schema + React Hook Form, 不用 v-model
12. 更新 AGENTS.md → 新增条目
```
