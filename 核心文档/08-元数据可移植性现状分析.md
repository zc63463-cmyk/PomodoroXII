# 元数据可移植性分析与改进方案

> 生成时间：2026-07-01 | 基于 React 19 技术栈重构 | 目标：让元数据对 Agent 和插件可读、可分析、可移植

---

## 一、现状诊断：元数据可移植性评分

### 1.1 核心发现：元数据几乎完全锁定在 DB 中

对 4 类实体的元数据存储方式做了逐一排查：

| 实体 | 元数据存储位置 | .md 文件含 frontmatter? | Agent 可直接读? |
|------|--------------|----------------------|----------------|
| Note | pomodoroxi.db (16 字段) | ❌ 纯正文 | ❌ 需 API |
| Task | pomodoroxi.db (14 字段) | ❌ 无 .md 文件 | ❌ 需 API |
| Session | pomodoroxi.db (16 字段含丰富指标) | ❌ 无 .md 文件 | ❌ 需 API |
| QuickNote (已归档) | pomodoroxi.db + .md frontmatter | ✅ archive.py 写了 frontmatter | ✅ 可读 |

**关键发现**：`archive.py` 是项目中**唯一**写了 YAML frontmatter 的地方（第 55-64 行），但 file_system 子系统的 `create_note` 写 .md 时**只写纯正文，没有 frontmatter**。

**前端类型层分析**：React 项目中的前端 TypeScript 类型定义位于 `frontend/src/types/index.ts`，该文件原样从 pomodoroxi 迁移（框架无关），其中各实体 interface 同样将同步基础设施字段（`_dirty`/`_etag`/`synced`/`content_hash`）与用户业务字段混合定义在同一类型中。Agent 即使读取前端类型定义，也无法从类型层面区分用户数据与同步元数据。

### 1.2 六个可移植性障碍

| # | 障碍 | 现状 | 对 Agent 的影响 |
|---|------|------|----------------|
| 1 | .md 无 frontmatter | file_system 写纯正文，元数据全在 index.db | Agent 读 .md 只看到正文，不知道 tags/folder/时间 |
| 2 | 时间戳非标准 | `String(32)` 存 ISO 无 Z 后缀 | Agent 需额外 normalize，易出错 |
| 3 | tags 存为 JSON 字符串 | `tags: Mapped[str] = "['a','b']"` | 不可 SQL 查询，Agent 需手动 parse |
| 4 | 关系隐式 | task_id/session_id/folder_id 是逻辑 FK | Agent 无法从单文件推断实体关系图 |
| 5 | 同步元数据混入业务数据 | _dirty/_etag/updated_at 与用户数据同表；前端 `frontend/src/types/index.ts` 中同步字段与业务字段同 interface | Agent 拿到一堆无关字段 |
| 6 | 无批量导出 API | export.ts 只生成人类可读报告 | Agent 无法获取结构化全量数据 |

### 1.3 现有 frontmatter 模式（archive.py — 唯一的好榜样）

```python
# archive.py 第 55-71行 — 已有的正确模式
frontmatter = {
    "id": note["id"],
    "title": note.get("content", "").split("\n")[0][:50] or "无标题",
    "mood": note.get("mood"),
    "tags": note.get("tags", []),
    "pinned": note.get("pinned", False),
    "created_at": note.get("created_at"),
    "updated_at": note.get("updated_at"),
    "archived_at": note.get("archived_at"),
}
body = note.get("content", "")
file_content = f"---\n{yaml.dump(frontmatter, ...)}---\n\n{body}"
```

**这个模式应该推广到所有 .md 文件**。

## 二、改进方案

### 2.1 核心原则：文件自描述 + API 结构化 + 关系显式化

```
【目标】Agent / 插件可以通过三种方式获取元数据：

路径 A：直接读 .md 文件 → YAML frontmatter 包含完整元数据
路径 B：调用 API → 结构化 JSON，支持过滤和分页  
路径 C：批量导出 → JSONL 全量快照，离线分析

三条路径的数据必须一致，以 .md frontmatter 为 source of truth。
```

### 2.2 改进 1：.md 文件增加 YAML frontmatter

**现状**：file_system 的 create_note 只写纯正文：
```python
self._atomic_write(abs_path, content)  # 纯正文，无 frontmatter
```

**改进**：写入时附加 frontmatter：

```python
# engine/note_ops.py — create_note / edit_note 修改
def _build_md_content(self, note_id: str, title: str, content: str,
                      tags: list[str], folder_id: str | None,
                      created_at: str, updated_at: str,
                      content_hash: str, word_count: int) -> str:
    """构建带 frontmatter 的 .md 文件内容"""
    frontmatter = {
        "id": note_id,
        "title": title,
        "tags": tags,                    # 原生 list，非 JSON 字符串
        "folder_id": folder_id,
        "content_hash": content_hash,
        "word_count": word_count,
        "created_at": created_at,        # 标准化：带 Z 后缀
        "updated_at": updated_at,
    }
    return f"---\n{yaml.dump(frontmatter, allow_unicode=True, default_flow_style=False)}---\n\n{content}"
```

**效果**：Agent 读到的 .md 文件：

```markdown
---
id: n_abc123
title: 深度工作笔记
tags:
  - 专注力
  - 方法论
folder_id: f_work
content_hash: sha256xxx
word_count: 1200
created_at: 2026-07-01T10:00:00.000Z
updated_at: 2026-07-01T11:30:00.000Z
---

今天尝试了深度工作法，连续 4 个番茄钟...
```

Agent 无需 DB 就能获取笔记的全部元数据。

### 2.4 改进 3：时间戳标准化

**现状**：`String(32)` 存 `"2026-07-01T10:00:00.123456"`（无 Z 后缀）

**问题**：Agent 拿到时间戳不知道时区，需要额外的 normalize 逻辑

**改进**：导出/frontmatter 中统一用 ISO 8601 UTC 带后缀：

```python
# services/time.py 新增
def utc_now_iso_z() -> str:
    """返回带 Z 后缀的标准 ISO 8601 时间戳"""
    return datetime.now(timezone.utc).strftime("%Y-%m-%dT%H:%M:%S.") + \
           f"{datetime.now(timezone.utc).microsecond // 1000:03d}Z"

# frontmatter / export 中使用
"created_at": utc_now_iso_z(),  # "2026-07-01T10:00:00.123Z"
```

**注意**：DB 内部存储保持现状（无 Z，用于 LWW 词法比较），仅在导出/frontmatter 时转换。

### 2.5 改进 4：关系图导出

**现状**：task→session、note→folder、note→note_links 都是隐式 FK 引用

**改进**：导出显式关系图，供 Agent 做关联分析：

```jsonl
// data/export/relations.jsonl
{"from":"task_001","to":"session_abc","type":"has_session"}
{"from":"task_001","to":"note_xyz","type":"referenced_in"}
{"from":"note_xyz","to":"folder_work","type":"belongs_to"}
{"from":"note_abc","to":"note_xyz","type":"links_to","context":"参见深度工作笔记"}
```

Agent 可以构建知识图谱，分析"哪些任务引用了哪些笔记"、"哪些笔记之间有链接"。

### 2.6 改进 5：用户数据与同步元数据分离

**现状**：`_dirty`, `_etag`, `synced`, `updated_at` 混在一起

**改进**：导出时剥离同步元数据，只保留用户数据：

```python
SYNC_INFRA_FIELDS = {"_dirty", "_etag", "synced", "content_hash"}

def to_user_facing_dict(obj) -> dict:
    """导出给 Agent / 插件的纯净用户数据"""
    data = {c.name: getattr(obj, c.name) for c in obj.__table__.columns}
    for field in SYNC_INFRA_FIELDS:
        data.pop(field, None)
    # 标准化时间戳
    for ts_field in ("created_at", "updated_at", "started_at", "ended_at"):
        if ts_field in data and data[ts_field]:
            data[ts_field] = normalize_to_iso_z(data[ts_field])
    # 解析 JSON 字符串字段
    if "tags" in data and isinstance(data["tags"], str):
        data["tags"] = json.loads(data["tags"])
    return data
```

**前端类型配合**：React 项目 `frontend/src/types/index.ts` 中应将同步基础设施字段拆分到独立 `SyncMeta` interface，业务类型通过 `Omit<FullEntity, SyncFieldKeys>` 派生纯净用户类型，使 Agent 从前端类型层即可区分用户数据与同步基础设施：

```typescript
// frontend/src/types/index.ts — 类型拆分改进
interface SyncMeta {
  _dirty: boolean
  _etag: string | null
  synced: boolean
  content_hash: string
}

// 完整实体（含同步字段，内部使用）
interface NoteFull extends SyncMeta {
  id: string
  title: string
  content: string
  tags: string[]
  folder_id: string | null
  created_at: string
  updated_at: string
}

// 用户可见类型（Agent / 插件消费）
type Note = Omit<NoteFull, keyof SyncMeta>
```

| 关系显式 | ❌ 隐式 FK | ✅ relations.jsonl | 质变 |
| 批量导出 | ❌ 仅人类报告 | ✅ JSONL + API | 质变 |

| Agent 直接读文件 | ❌ 需 API | ✅ 读 .md 即可 | 质变 |
| 插件扩展 | ❌ 无接口 | ✅ query API + export | 质变 |
