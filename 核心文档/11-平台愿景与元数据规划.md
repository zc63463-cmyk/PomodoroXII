## 一、愿景：个人状态管理平台

### 1.1 一句话描述

为**深度自我管理者**解决**个人状态数据分散、无法跨维度关联分析**的问题，PomodoroXII 提供一个**以番茄钟为核心引擎的个人元数据产出平台**，产出**时间/专注/认知/行为四维元数据**，与 MarkVault 的知识标注元数据融合，形成"知识获取（MarkVault）→ 时间投入（PomodoroXII）→ 状态产出（元数据分析）"的闭环。

### 1.2 两个项目的定位关系

```
┌─────────────────────────────────────────────────────────────┐
│  个人状态管理平台                                            │
│                                                              │
│  ┌──────────────────┐         ┌──────────────────┐         │
│  │  MarkVault        │         │  PomodoroXII      │         │
│  │  (知识层面)        │         │  (管理层)          │         │
│  │                   │         │                   │         │
│  │  标注元数据        │←──────→│  状态元数据        │         │
│  │  • 学习了什么      │  元数据  │  • 何时学的        │         │
│  │  • 理解程度        │  融合    │  • 专注多久        │         │
│  │  • 知识图谱        │         │  • 认知状态        │         │
│  │  • 标注/批注       │         │  • 行为模式        │         │
│  │                   │         │                   │         │
│  │  深度：知识结构    │         │  深度：状态分析    │         │
│  └──────────────────┘         └──────────────────┘         │
│                     │                                       │
│                     ▼                                       │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  元数据融合层 (后续深化)                                │  │
│  │  • "学某类知识时的专注力曲线"                           │  │
│  │  • "知识吸收效率与时间段的关系"                         │  │
│  │  • "学习行为模式 → 个性化推荐"                          │  │
│  │  • Agent 可读的统一元数据 API                          │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 PCV 快速评估

| 维度 | 0 分 | 1 分 | 2 分 | 得分 |
|------|------|------|------|------|
| 痛点真实度 | 臆想 | 存在但不频繁 | 高频且痛苦 | **2** — 深度自我管理者确实需要跨维度状态分析，目前没有任何工具能同时管时间+知识+认知 |
| 付费意愿 | 不愿 | 有意愿无预算 | 有预算已寻找 | **1** — 个人用户付费意愿中等，知识管理赛道有付费先例（Notion/Obsidian Sync） |
| 竞争壁垒 | 无差异 | 有一些差异 | 有明确壁垒 | **2** — 番茄钟+笔记+复盘+认知标记+元数据融合的组合独一无二，MarkVault 标注系统是独有数据源 |
| 可执行性 | >3月 | 1-3月 | <2周可验证 | **1** — PomodoroXII 本身需 5-6 周开发，元数据融合是后续阶段 |
| 时机 | 太早/晚 | 一般 | 窗口期正开 | **2** — AI Agent 时代，个人元数据可被 Agent 消费是 2026 独有窗口 |

**总分：8/10** — 值得深入，但需分阶段执行。

### 1.4 关键判断：初级阶段不验证平台，验证元数据产出

**重要决策**：PomodoroXII 初级阶段（M0-M6）**不追求平台化**，而是确保**元数据产出的使用逻辑健全性和稳定性**。

理由：
- 没有稳定可靠的元数据产出，平台化是无源之水
- pomodoroxi 已有 38 个子功能的实战验证，PomodoroXII 是架构升级而非功能验证
- MarkVault 融合是后续阶段，当前只需确保 PomodoroXII 的元数据**可产出、可导出、可被 Agent 消费**
- MCP Server + Export API + .md frontmatter 三路径已经为平台化预留了接口

## 二、PomodoroXII 元数据产出体系

### 2.1 四维元数据定义

| 维度 | 数据来源 | 元数据字段 | Agent 消费场景 |
|------|---------|-----------|---------------|
| **时间** | 番茄钟会话 | started_at, ended_at, duration, type, completed | "我上周专注了多少小时" |
| **专注** | 认知标记 | attention_score, flow_state, distraction_count | "什么时间段我最专注" |
| **认知** | 复盘 + 事件 | mood, energy, achievements, challenges, learnings | "我的状态趋势如何" |
| **行为** | 任务/习惯/时间块 | task_completion_rate, habit_streak, time_block_distribution | "我的行为模式是什么" |

### 2.2 元数据产出路径

> 前端层：**React 19 + Next.js 15（App Router）+ Zustand 状态管理**（原 Vue 3 + Pinia 已替换为渐进迁移，详见 01 文档 M5a/M5b/M5c）

```
PomodoroXII 元数据产出
    │
    ├── 实时产出（用户操作触发 — React 19 前端 + Zustand 客户端状态）
    │   ├── 番茄钟会话 → session 元数据（时间+专注+认知）
    │   ├── 任务完成 → task 元数据（行为+时间）
    │   ├── 习惯打卡 → habit 元数据（行为+时间）
    │   ├── 复盘写入 → reflection 元数据（认知+session快照）
    │   └── 笔记创建 → note 元数据（知识+时间）
    │
    ├── 定时产出（自动快照）
    │   ├── 每日 → data/export/*.jsonl（全量快照）
    │   ├── 每日 → reflections 归档 .md（复盘文件化）
    │   └── 每小时 → 一致性检查（元数据完整性）
    │
    └── 按需产出（Agent/插件请求）
        ├── MCP Server → query_tasks / search_notes / get_focus_stats
        ├── REST API → /api/export/full (含 _graph + _analysis_hints)
        └── 文件读取 → .md frontmatter + AGENTS.md
```

### 2.3 元数据质量保障（初级阶段核心）

**这是初级阶段的重点**——不是做多少功能，而是确保产出的元数据可靠、一致、可消费。

| 质量维度 | 保障措施 | 对应文档 |
|---------|---------|---------|
| **完整性** | sync_safety.py 5 道安全检查 + 一致性检查定时任务 | 01 架构规划, 06 实施评审 |
| **一致性** | Saga 模式跨库一致性 + DB↔.md 定时修复 | 05 移植分析 |
| **可追溯** | sync_events 审计日志 + version 版本号 + .md 版本历史 | 04 数据管理分析 |
| **标准化** | 时间戳 DB 内无 Z + 导出加 Z + tags 原生 list | 08 可移植性现状 |
| **可消费** | MCP Server + Export API + .md frontmatter 三路径 | 07 可移植性升级 |
| **不丢失** | 双库自动备份 + Outbox 合并矩阵 + Tombstone 防复活 | 04 数据管理分析 |

### 2.4 与 MarkVault 的融合预留

PomodoroXII 在初级阶段不需要实现融合，但需要为融合预留接口：

| 预留点 | 实现方式 | 当前状态 |
|--------|---------|---------|
| **统一实体 ID** | PomodoroXII 用 UUID，MarkVault 标注也用 UUID，可通过 task_id/note_id 关联 | 已就绪 |
| **session_snapshot** | 复盘 frontmatter 含当日 session 快照，MarkVault 可读取"学习某知识时的专注状态" | 需求 1 已设计 |
| **content_hash** | note 的 content_hash 可被 MarkVault 引用，检测知识源变化 | 01 架构规划已设计 |
| **Export API** | /api/export/full 返回 _graph 关系图，MarkVault 可消费建立知识-时间关联 | 07 升级方案已设计 |
| **MCP Server** | query_tasks/get_focus_stats 工具可被 MarkVault 的 Agent 调用 | 07 升级方案已设计 |
| **relations.jsonl** | 定时快照含实体关系图，MarkVault 可读取 task→session→reflection 链路 | 07 升级方案已设计 |

## 三、分阶段路线图

### Phase 1：PomodoroXII 核心开发（M0-M6，~27 天）

**目标**：稳定产出四维元数据

| 里程碑 | 元数据产出能力 | 验证标准 |
|--------|--------------|---------|
| M0 骨架 | 项目结构 + 配置 | uvicorn 启动通过 |
| M1 认证+Task | task 元数据可产出 | CRUD + sync 通过 |
| M2 Sync 引擎 | 全实体元数据可同步 | push/pull + 墓碑 + 级联通过 |
| M3 file_system | note 元数据可文件化 | .md + frontmatter + FTS5 通过 |
| M4 可靠性+Agent | 元数据可备份+可导出+可被 Agent 读 | 备份 + 快照 + MCP 连接通过 |
| M5 前端 | 用户可交互产出元数据 | React 前端全功能跑通 |
| M6 部署 | 线上稳定产出 | 线上同步跑通 |

> **M5 拆分为三阶段渐进式重建**（详见 01 文档 Milestone 5，共 9 天）：
>
> | 阶段 | 内容 | 验证标准 |
> |------|------|---------|
> | **M5a** React 骨架 + 框架无关代码迁移（3 天） | `create-next-app` + App Router 初始化、Tailwind v4 + shadcn/ui 初始化、框架无关代码（services/types/utils/workers）原样迁移、Zustand store 骨架（17 个 `create()` 空壳）、`@serwist/next` 配置 | `next dev` 启动 + 空首页渲染 + Dexie 连接 + `app/` 路由生效 |
> | **M5b** 逐 View 迁移（4 天） | `composables/` → `hooks/`（40+ `useXxx`）、Pinia actions → Zustand store（17 个逻辑平移）、`views/*.vue` → `app/**/page.tsx`、React Hook Form + Zod 替代 v-model、TanStack Query v6 接入 | 各页面功能跑通 + 表单验证 + dark mode 切换 |
> | **M5c** 清理 Vue + PWA 收尾（2 天） | 删除 Vue 残留（App.vue / router/ 及 vue/vue-router/pinia 依赖）、PWA Service Worker 注册、React 19 Compiler 启用、Zustand store selector 单测、E2E 测试适配 | 前端全功能跑通 + PWA 离线可用 + 无 Vue 依赖 + Compiler 生效 |

### Phase 2：元数据质量加固（M6 后，1-2 周）

**目标**：确保元数据在长期运行中的可靠性

| 任务 | 说明 |
|------|------|
| 一致性检查实战 | 运行 1 周，检查 DB↔.md 不一致频率 |
| 备份恢复演练 | 模拟 DB 损坏，从备份+文件恢复 |
| sync 压力测试 | 大量事件并发 push，验证 SAVEPOINT |
| 导出完整性验证 | 对比 DB 数据与 JSONL 快照 |
| 归档自动化验证 | 30 天后复盘自动归档 |

### Phase 3：MarkVault 融合（Phase 2 后，2-4 周）

**目标**：知识元数据 + 状态元数据融合

| 任务 | 说明 |
|------|------|
| 定义融合接口 | PomodoroXII 的 task_id/note_id 与 MarkVault 的标注 ID 关联规范 |
| 统一 Export | PomodoroXII Export API + MarkVault 标注 API → 统一 JSONL |
| 统一 MCP | 一个 MCP Server 暴露两个项目的 Tools/Resources |
| 关联分析 | "学某类知识时的专注力曲线"——跨项目 JOIN |

### Phase 4：平台化（远期，3-6 月）

**目标**：个人状态管理平台

| 能力 | 说明 |
|------|------|
| 跨项目知识-时间图谱 | MarkVault 知识节点 + PomodoroXII 时间节点 → 统一图 |
| Agent 自动分析 | MCP Server 暴露全量元数据，Agent 自动生成洞察 |
| 个性化推荐 | 基于历史元数据推荐"最佳学习时段""最佳任务类型" |
| 多设备同步 | 已有 sync 引擎支持 |

## 四、初级阶段的"不可做"清单

为了保护元数据产出的稳定性，初级阶段明确不做以下事情：

| 不做 | 理由 |
|------|------|
| ~~多用户支持~~ | 单用户场景，user_id 隔离增加复杂度但不解决实际问题 |
| ~~实时 MarkVault 融合~~ | 融合是 Phase 3，当前只需预留接口 |
| ~~AI 自动分析~~ | Agent 消费是 P4，当前只需确保元数据可被 Agent 读 |
| ~~CRDT 协同编辑~~ | 单用户离线同步，LWW + content_hash 足够 |
| ~~PostgreSQL~~ | SQLite WAL 单用户够用，换库增加运维 |
| 前端框架替换是既定决策，但采用渐进迁移策略 | Vue 3 → React 19 的替换已确定，通过 M5a/M5b/M5c 三阶段渐进迁移（非大爆炸重写），框架无关代码（services/types/utils/workers 约 30%）直接复用 |
