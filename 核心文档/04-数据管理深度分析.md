## 一、项目现状诊断

### 1.1 整体架构画像

PomodoroXI 采用 **Client-first 离线同步架构**：

- **前端 IndexedDB（Dexie.js）** 是本地数据的唯一真相源
- **后端 SQLite** 是同步副本，提供跨设备数据一致性
- **同步协议**：pull/push 双向增量同步，LWW 冲突解决，Tombstone 防删除复活

```
┌─────────────────────────────────────────────────────┐
│  Client (React 19 + Dexie.js)                       │
│  ┌──────────┐  ┌──────────┐  ┌───────────────────┐  │
│  │ Zustand  │←→│ Outbox   │←→│ IndexedDB (v15)   │  │
│  │ Stores   │  │ Events   │  │ 16 tables          │ │
│  └──────────┘  └────┬─────┘  └───────────────────┘  │
│                     │                               │
│              ┌───────┴───────┐                      │
│              │ useSync hooks │ pull/push/fullSync   │
│              │ (3 modules)   │ LWW merge + tombstone│
│              └───────┬───────┘                      │
│              (hooks, sync logic preserved)          │
└─────────────────────┼───────────────────────────────┘
                      │ HTTP (JWT)
┌─────────────────────┼───────────────────────────────┐
│  Server (FastAPI)   │                                │
│              ┌──────┴──────┐                         │
│              │ sync.py     │ 1047 lines              │
│              │ ENTITY_REG  │ 14 entity types          │
│              └──────┬──────┘                         │
│  ┌──────────┐  ┌────┴─────┐  ┌───────────────────┐ │
│  │ REST API │  │ Tombstone│  │ SQLite (WAL mode) │ │
│  │ 13 routes│  │ TTL 90d  │  │ 16 tables          │ │
│  └──────────┘  └──────────┘  └───────────────────┘ │
└─────────────────────────────────────────────────────┘
```

| Setting | settings | (不同步) |
| Tombstone | tombstones | deleted_at |

**关键设计特征**：

1. **时间戳全部用 `String(32)` 存储** ISO 8601 字符串（无 Z 后缀），利用 SQLite 词法字符串比较做 LWW 排序
2. **无 `etag`/`version`/`revision` 字段** — 同步元数据纯靠 `updated_at` 字符串比较
3. **无 `user_id` 字段** — 单用户模式，所有查询全局可见
4. **无数据库级外键约束** — 级联删除依赖应用代码
5. **JSON 数组字段存为字符串** — tags、sections、related_task_ids 等

### 1.3 同步引擎现状

#### 后端 sync.py（1047 行，项目最复杂的单文件）

**核心机制**：

| 机制 | 实现 | 评价 |
|------|------|------|
| 增量拉取 | `WHERE updated_at > since` + 分页 | 基本可用 |
| 批量推送 | POST events[], max 500/批, SAVEPOINT 隔离 | 设计合理 |
| 冲突解决 | LWW（字符串比较 updated_at） | 依赖客户端时钟，有风险 |
| 防删除复活 | Tombstone + created_at 比较 + TTL 90d 保护 | 多层防护，较完善 |
| 零点时间戳检测 | 识别 `00:00:00.000` 为陈旧数据 | 生产经验体现 |
| 级联删除 | task→sessions+quicknotes, habit→checkins, folder→BFS | 逻辑分散在两处 |
| Schema 迁移 | 自定义 `_migrate_schema`，只支持 ADD COLUMN | **严重不足** |

#### 前端同步引擎（三模块拆分）

| 模块 | 职责 | 文件 |
|------|------|------|
| useSync.ts | Facade 门面，统一 state | hook |
| useSyncCloud.ts | pull/push/fullSync + LWW merge | 1193 行 |
| useSyncOutbox.ts | 事件入队 + 合并矩阵 | database.ts 内 |

**Outbox 合并矩阵**（前端最成熟的部分）：

```
┌──────────┬──────────┬──────────┬──────────┐
│ existing │ create   │ update   │ delete   │
├──────────┼──────────┼──────────┼──────────┤
│ create   │ create*  │ create↗  │ ⊘ DROP   │
│ update   │ update*  │ update↗  │ delete↗  │
│ delete   │ update↗  │ delete   │ delete   │
└──────────┴──────────┴──────────┴──────────┘
↗ = 取新 payload   ⊘ = 移除事件+本地实体   * = 不应发生
```

> 注：useSync 三模块从 Vue composables 迁移为 React hooks，pull/push/fullSync + LWW merge + tombstone 同步逻辑保留不变。

## 三、行业参考方案对比

### 3.1 离线优先同步引擎全景

基于 2026 年最新行业调研，离线优先同步引擎分为三大流派：

| 流派 | 代表方案 | 冲突解决 | 数据模型 | 适用场景 |
|------|---------|---------|---------|---------|
| **服务器权威 + Mutator** | Replicache, Zero, PowerSync | 服务器决定最终状态 | 关系型 SQL | 每用户编辑自己数据 |
| **CRDT** | Y.js, Automerge, Loro | 数学保证自动合并 | 文档/JSON | 多用户协同编辑 |
| **事件溯源** | LiveStore | 事件排序决定一致性 | 事件日志 | 需要审计/时间旅行 |
| **每列 LWW + 约束** | ElectricSQL | 细粒度字段级 LWW | 关系型 SQL | Postgres + SQLite |

### 3.2 与 PomodoroXI 最相关的参考方案

#### 方案 A：PowerSync（最接近的技术路线）

**为什么相关**：PowerSync 同样是 SQLite 客户端 + 后端数据库同步，架构与 PomodoroXI 高度相似。

**PowerSync 的核心设计**：

1. **Bucket 系统**：数据按逻辑分桶（bucket），每个客户端只同步自己有权访问的桶。增量同步基于桶内操作历史（operation history），而非全表扫描。

2. **Sync Streams（新）/ Sync Rules（旧）**：用 SQL-like 语法定义同步范围，支持参数化（JWT 中的 user_id 等），实现行级权限过滤。

3. **Write Checkpoints**：客户端写入后获得 checkpoint ID，服务端重放 mutation 时用它做冲突检测——比纯 LWW 更可靠。

4. **客户端 Schema 管理**：客户端 SQLite 用 `PRAGMA user_version` 管理 schema 版本，提供与服务器端相同的迁移模式。

5. **复制机制**：服务端通过 Postgres logical replication 拉取 WAL 变更，转换为 bucket 操作，增量推送给客户端。

**可借鉴点**：
- Bucket 分桶同步（替代全表 `WHERE updated_at > since`）
- Write Checkpoints（替代纯 LWW 时钟比较）
- `PRAGMA user_version` schema 版本管理
- 声明式同步规则

#### 方案 B：Replicache（Mutator 模式参考）

**为什么相关**：Replicache 的 Mutator 模式可以改进 PomodoroXI 的 outbox + push 机制。

**Replicache 的核心设计**：

1. **Mutator 纯函数**：每个变更操作定义为纯函数（如 `createTodo(tx, args)`），在客户端乐观执行，服务端重放。客户端和服务端实现同一函数。

2. **client ID + last-mutation-ID**：服务端跟踪每个客户端已处理的最新 mutation ID，实现精确的增量同步。

3. **Patch 机制**：服务端返回 diff 作为 patch，应用到客户端缓存，而非返回完整对象。

**可借鉴点**：
- Mutator 模式可以让级联删除逻辑在客户端和服务端一致执行
- client ID + mutation ID 比 `updated_at` 字符串比较更可靠

#### 方案 C：Event Sourcing / LiveStore（元数据管理参考）

**为什么相关**：PomodoroXI 的 outbox 已经是事件溯源的雏形，可以进一步演进。

**LiveStore 的核心设计**：

1. **事件日志为核心**：每个变更记录为不可变事件，物化视图从事件流派生
2. **时间旅行**：回放事件到任意点，天然支持审计
3. **Schema 演进优势**：只需保证事件 schema 兼容，视图可自由重新派生

**可借鉴点**：
- 将 outbox 从临时队列升级为持久化事件日志
- 物化视图可从事件流重新派生，降低 schema 迁移风险

#### 方案 D：HLC（混合逻辑时钟）替代纯 LWW

**为什么相关**：解决 P1-3（LWW 依赖客户端时钟）的最佳方案。

**HLC 核心原理**：

- 时间戳 = `(物理时间 l, 逻辑计数器 c)`，编码为 64 位整数
- `l = max(本地旧值, 当前物理时间)` — 容忍时钟倒退
- `c` 在同一物理时间时递增 — 保证单调性
- 比较规则：先比物理时间，相同则比计数器

**HLC vs 纯墙钟 LWW**：

| 维度 | 纯墙钟 LWW | HLC |
|------|-----------|-----|
| 因果正确性 | 无（时钟倒退会错误覆盖） | 有（逻辑计数器保证单调） |
| 人类可读 | 是 | 是（物理部分接近墙钟） |
| 实现复杂度 | 低 | 中 |
| 时钟偏差容忍 | 无 | 有（max 函数 + 计数器兜底） |
| 生产验证 | 基础方案 | CockroachDB, MongoDB, YugabyteDB |

### 3.3 决策矩阵

| 维度 | PowerSync | Replicache | LiveStore | HLC 改进 | 保持现状 |
|------|-----------|------------|-----------|---------|---------|
| 与现有架构兼容性 | 中 | 中 | 低 | **高** | 高 |
| 迁移成本 | 高 | 高 | 极高 | **低** | 无 |
| 解决 P0 问题 | 部分 | 部分 | 部分 | 否 | 否 |
| 解决 P1 问题 | 是 | 是 | 是 | **是** | 否 |
| 长期可维护性 | 高 | 高 | 高 | 中 | 低 |
| 学习曲线 | 中 | 中 | 高 | **低** | 无 |

**推荐策略**：渐进式改进，优先采用 **HLC + Alembic + 服务层统一 + 事件日志持久化**，而非整体替换为 PowerSync/Replicache。

## 四、改进方案设计

### 4.1 元数据管理规范化

#### 4.1.1 引入版本号字段（替代纯 updated_at LWW）

**现状**：所有模型只有 `created_at` + `updated_at`（ISO 字符串），LWW 纯靠字符串比较。

**改进**：每个可同步模型增加 `version` 字段（整数递增）：

```python
# backend/app/models/base.py（新建）
class SyncMixin:
    """可同步实体的基类 Mixin"""
    created_at: Mapped[str] = mapped_column(String(32), default=utc_now_iso)
    updated_at: Mapped[str] = mapped_column(String(32), default=utc_now_iso)
    version: Mapped[int] = mapped_column(Integer, default=1)  # 新增
    deleted_at: Mapped[str | None] = mapped_column(String(32), nullable=True)  # 软删除统一
```

**冲突解决改进**：

```python
# 旧：纯 updated_at 字符串比较
if server_time > client_time:
    conflicts.append(...)

# 新：version + updated_at 双重比较
if server_version > client_version:
    conflicts.append({"reason": "server_version_newer", ...})
elif server_version == client_version and server_time > client_time:
    # 同版本但时间更新（理论不应发生，防御性）
    conflicts.append({"reason": "time_divergence", ...})
```

#### 4.1.2 HLC 时间戳（可选进阶）

若要彻底解决时钟偏差问题，引入 HLC：

```python
# backend/app/services/hlc.py（新建）
import threading
from datetime import datetime, timezone

class HLC:
    """Hybrid Logical Clock implementation"""
    def __init__(self):
        self._l = 0  # physical time (ms)
        self._c = 0  # logical counter
        self._lock = threading.Lock()

    def now(self) -> int:
        """Generate a new HLC timestamp (64-bit int)"""
        with self._lock:
            pt = int(datetime.now(timezone.utc).timestamp() * 1000)
            if pt > self._l:
                self._l = pt
                self._c = 0
            else:
                self._c += 1
            # 48-bit physical + 16-bit counter = 64-bit
            return (self._l << 16) | (self._c & 0xFFFF)

    def receive(self, remote_hlc: int):
        """Update local HLC upon receiving a remote timestamp"""
        with self._lock:
            remote_l = remote_hlc >> 16
            remote_c = remote_hlc & 0xFFFF
            pt = int(datetime.now(timezone.utc).timestamp() * 1000)
            new_l = max(self._l, remote_l, pt)
            if new_l == self._l == remote_l:
                self._c = max(self._c, remote_c) + 1
            elif new_l == self._l:
                self._c += 1
            elif new_l == remote_l:
                self._c = remote_c + 1
            else:
                self._c = 0
            self._l = new_l

hlc = HLC()
```

**存储方式**：`hlc_timestamp` 字段用 `BIGINT`（64 位整数），替代 `String(32)` 的 `updated_at`。

#### 4.3.2 Tombstone 服务化

```python
# backend/app/services/tombstone.py（新建）
class TombstoneService:
    @staticmethod
    async def create(db: AsyncSession, entity_type: str, entity_id: str):
        """创建或更新 tombstone"""
        tombstone = Tombstone(
            entity_type=entity_type,
            entity_id=entity_id,
            deleted_at=utc_now_iso()
        )
        await db.merge(tombstone)  # upsert

    @staticmethod
    async def exists(db: AsyncSession, entity_type: str, entity_id: str) -> bool:
        """检查 tombstone 是否存在"""
        result = await db.execute(
            select(Tombstone).where(
                Tombstone.entity_type == entity_type,
                Tombstone.entity_id == entity_id
            )
        )
        return result.scalar_one_or_none() is not None

    @staticmethod
    async def cleanup_expired(db: AsyncSession, ttl_days: int = 90):
        """清理过期 tombstone — 改为独立定时任务，不在 pull 中执行"""
        cutoff = (datetime.utcnow() - timedelta(days=ttl_days)).isoformat()
        await db.execute(delete(Tombstone).where(Tombstone.deleted_at < cutoff))
```

### 4.5 数据库备份策略

```python
# backend/app/services/backup.py（新建）
import shutil
from pathlib import Path
from datetime import datetime

class BackupService:
    DB_PATH = "data/pomodoroxi.db"
    BACKUP_DIR = "data/backups"

    @classmethod
    async def create_backup(cls) -> str:
        """使用 SQLite Online Backup API 创建备份"""
        timestamp = datetime.utcnow().strftime("%Y%m%d_%H%M%S")
        backup_path = f"{cls.BACKUP_DIR}/pomodoroxi_{timestamp}.db"

        # SQLite Online Backup（不锁库）
        async with aiosqlite.connect(cls.DB_PATH) as db:
            await db.execute(f"BACKUP DATABASE TO '{backup_path}'")
            await db.commit()

        # 清理旧备份（保留最近 30 天）
        cls._cleanup_old_backups(30)
        return backup_path

    @classmethod
    def _cleanup_old_backups(cls, keep_days: int):
        """清理超过 keep_days 天的备份"""
        cutoff = datetime.utcnow().timestamp() - (keep_days * 86400)
        for f in Path(cls.BACKUP_DIR).glob("pomodoroxi_*.db"):
            if f.stat().st_mtime < cutoff:
                f.unlink()
```

### 5.3 Service Layer 设计

```python
# backend/app/services/base.py
class BaseService:
    """所有业务 Service 的基类"""
    def __init__(self, db: AsyncSession):
        self.db = db

    async def get(self, model_cls, entity_id: str):
        result = await self.db.execute(
            select(model_cls).where(model_cls.id == entity_id)
        )
        return result.scalar_one_or_none()

    async def create(self, model_cls, data: dict):
        obj = model_cls(**data)
        obj.version = 1
        self.db.add(obj)
        await self.db.flush()
        return obj

    async def update(self, obj, data: dict):
        for key, value in data.items():
            setattr(obj, key, value)
        obj.version += 1  # 版本号递增
        obj.updated_at = utc_now_iso()
        await self.db.flush()
        return obj

    async def delete(self, obj, cascade: bool = True):
        if cascade:
            await self._cascade_delete(obj)
        await self.db.delete(obj)
```

```python
# backend/app/services/task_service.py
class TaskService(BaseService):
    async def create_task(self, data: dict) -> Task:
        return await self.create(Task, data)

    async def update_task(self, task_id: str, data: dict) -> Task:
        task = await self.get(Task, task_id)
        if not task:
            raise NotFoundError(f"Task {task_id} not found")
        return await self.update(task, data)

    async def delete_task(self, task_id: str):
        task = await self.get(Task, task_id)
        if not task:
            raise NotFoundError(f"Task {task_id} not found")
        await CascadeService(self.db).delete_task(task_id)
        await TombstoneService(self.db).create("task", task_id)

    async def list_tasks(self, filters: dict, pagination: dict):
        query = select(Task)
        if filters.get("status"):
            query = query.where(Task.status == filters["status"])
        # ... 其他过滤
        query = query.offset(pagination["offset"]).limit(pagination["limit"])
        result = await self.db.execute(query)
        return result.scalars().all()
```

    user = Depends(get_current_user),
