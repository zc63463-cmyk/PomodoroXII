# 单用户多空间架构设计

> 生成时间：2026-07-02 | 基于 React 19 技术栈重构 | 参考 Obsidian Vault / Firefox Containers / VS Code Workspaces / Notion Workspace / Auth0 Organizations
> 定位：在 PomodoroXII 现有单用户架构基础上，扩展为多空间模式，实现类似隐私空间的独立数据隔离

---

## 一、设计目标与范围

### 1.1 需求背景

PomodoroXII 当前为单用户单空间架构：一个密码、一个数据库、一组数据。用户希望在不改变单用户本质的前提下，拥有多个独立"空间"（如"日常"、"测试"、"隐私"），每个空间：

- 拥有**完全隔离的数据**（任务/会话/笔记/速记/习惯/时间块等全部业务数据）
- 拥有**独立的同步通道**（outbox/syncMeta/tombstone 各自独立）
- 支持**可选的空间密码**（隐私空间可额外加锁）
- 支持**空间级别的配置覆盖**（如不同空间使用不同的专注时长）

### 1.2 设计目标

| 目标 | 衡量标准 |
|------|---------|
| 数据隔离 | 空间间数据物理隔离，不存在跨空间泄露风险 |
| 最小改动 | 16 张业务表 schema 不变，现有查询逻辑不变 |
| 空间切换 | < 500ms 完成空间切换（含 DB 连接切换 + UI 刷新） |
| 安全性 | 主密码保护空间列表，隐私空间支持额外密码 |
| 向后兼容 | 现有单空间数据自动迁移为默认空间，用户无感升级 |

### 1.3 不做范围（YAGNI）

- **不做跨空间功能**：无跨空间查询、聚合、复制、移动
- **不做多用户协作**：仍为单用户，空间不共享
- **不做空间级权限控制**：空间内所有数据平等，无细粒度权限
- **不做空间模板**：不预置空间模板，用户自行创建

### 1.4 用户决策（已确认）

| # | 决策项 | 选择 | 理由 |
|---|--------|------|------|
| 1 | 数据隔离方式 | **混合模式** | 共享 FastAPI 应用 + 每空间独立 SQLite/Dexie DB |
| 2 | 空间安全 | **主密码 + 可选空间密码** | 主密码保护空间列表，隐私空间可额外加锁 |
| 3 | 同步策略 | **每空间独立同步** | 独立 outbox/syncMeta/tombstone，隐私空间可关闭同步 |
| 4 | 跨空间功能 | **不需要** | 空间间完全隔离 |

## 二、成熟项目参考

深度研究了 7 种成熟的多空间/多工作空间架构模式：

### 2.1 Obsidian Vault 模式

Obsidian 的 Vault 是磁盘上的独立文件夹，包含所有 Markdown 笔记 + `.obsidian/` 配置目录。每个 Vault 完全独立，无共享数据。

| 维度 | Obsidian 实现 | 本设计借鉴 |
|------|-------------|-----------|
| 数据隔离 | 每个 Vault = 独立文件夹 | 每空间 = 独立 SQLite 文件 + 独立 Dexie DB |
| 配置隔离 | `.obsidian/` 目录存储 UI 状态/插件配置 | 每空间 DB 的 `settings` 表存储空间级配置 |
| 切换机制 | "Open another vault" 打开另一个文件夹 | SpaceSwitcher 下拉切换 |
| 同步 | 文件系统级同步（Git/Dropbox/iCloud）| 每空间独立 API 同步通道 |

**核心启示**：**文件夹即隔离边界**。映射到 SQLite，每个 Space 对应独立的 `.db` 文件。

### 2.2 Firefox Multi-Account Containers 模式

Firefox 通过 `OriginAttributes` 中的 `userContextId` 扩展同源策略，为每个容器分配独立的 Cookie/localStorage/IndexedDB 切片。

| 维度 | Firefox 实现 | 本设计借鉴 |
|------|-------------|-----------|
| 隔离机制 | Origin 四元组 `(scheme, host, port, userContextId)` | JWT 中注入 `space_id` 作为数据访问的隐式过滤条件 |
| 认证共享 | 密码/书签/历史跨容器共享 | 主密码跨空间共享，业务数据隔离 |

**核心启示**：在数据访问层注入 `space_id` 作为隐式过滤条件。但不同于 Firefox 的逻辑隔离（同一存储按 ID 分片），本设计采用物理隔离（独立 DB 文件），更安全。

### 2.3 VS Code Workspaces 模式

VS Code Workspace 支持配置继承 + 覆盖：Default Settings < Global User Settings < Workspace Settings < Folder Settings。

| 维度 | VS Code 实现 | 本设计借鉴 |
|------|-------------|-----------|
| 配置层级 | 四层继承，后者覆盖前者 | 全局设置（meta_settings）+ 空间级设置（space DB settings） |
| 扩展管理 | 按 Workspace 启用/禁用扩展 | 不适用（单应用无扩展系统） |
| 状态持久化 | `.code-workspace` JSON 文件 | `spaces` 表存储空间元数据 |

**核心启示**：配置分层。全局设置（主题、语言）所有空间共享，空间级设置（专注时长、提醒规则）各自独立。

### 2.4 Notion/Slack Workspace 模式

Notion 和 Slack 的 Workspace 切换通过 JWT 中的 `workspace_id` / `team_id` claim 实现。

| 维度 | Notion/Slack 实现 | 本设计借鉴 |
|------|------------------|-----------|
| JWT 绑定 | Access Token 包含 workspace_id | 双 JWT：Master Token + Space Token（含 space_id） |
| API 校验 | 验证 Token 中的 workspace_id 与请求匹配 | Space Token 中的 space_id 决定路由到哪个 DB |
| 切换机制 | 侧边栏切换器，切换时重新加载全部数据 | SpaceSwitcher 切换，重新加载 Dexie + Zustand store |

**核心启示**：JWT 携带空间标识，API 层强制校验。但 Notion/Slack 使用共享 DB + workspace_id 列，本设计使用独立 DB（更强隔离）。

### 2.5 Azure 多租户模式 C（混合模式）

Microsoft Azure 架构指南定义了三种多租户隔离模式：

| 模式 | 描述 | 隔离强度 | 管理复杂度 |
|------|------|---------|-----------|
| A. 共享 DB + tenant_id 列 | 所有租户共享 DB，查询加 WHERE tenant_id | 低（遗漏 WHERE 即泄露） | 低 |
| B. 每租户独立 DB | 每租户一个 DB 文件 | 高（物理隔离） | 高（迁移需遍历） |
| **C. 混合模式** | **共享应用 + 每租户独立 DB** | **高** | **中** |

本设计选择**模式 C（混合模式）**：
- 共享层：单一 FastAPI 实例 + Meta DB（`meta.db`）
- 隔离层：每空间独立 SQLite 文件（`space_{id}.db`）
- 路由层：SpaceEngineManager 动态引擎池

### 2.6 Auth0 Organizations 模式

Auth0 的 Organizations 功能在 JWT 中注入 `org_id` claim 和 `permissions` 数组。

| 维度 | Auth0 实现 | 本设计借鉴 |
|------|-----------|-----------|
| Token 结构 | `{ org_id, permissions }` | `{ type: "space", space_id }` |
| 组织切换 | 认证时传入 organization 参数 | `POST /auth/switch` 传入 space_id |
| 权限控制 | 每组织可有不同角色/权限 | 当前不需要（单用户无角色），预留扩展 |

**核心启示**：双 Token 模式——Master Token 管理组织列表，Scoped Token 绑定特定组织。

### 2.7 Dexie.js 多数据库能力

基于 Dexie.js 官方文档的研究：

| 能力 | 支持 | 说明 |
|------|------|------|
| 创建多个独立 DB | ✅ | `new Dexie("space_1")` / `new Dexie("space_2")` |
| 动态 DB 名 | ✅ | 数据库名是运行时字符串 |
| 删除 DB | ✅ | `Dexie.delete('space_1')` |
| 独立 schema version | ✅ | 每个 DB 有自己的 version 链 |
| **跨 DB 事务** | ❌ | 不支持跨 Dexie 实例的事务 |
| IndexedDB 配额 | 共享 | 所有 DB 共享同一 origin 的存储配额（磁盘 50%） |
| LRU 驱逐风险 | 存在 | 临时存储模式下可能被浏览器驱逐，需 `navigator.storage.persist()` |

**核心启示**：Dexie 原生支持多 DB，但需注意跨 DB 事务不支持 + 配额共享 + LRU 驱逐风险。

## 四、Meta 数据库设计

### 4.1 概述

Meta DB 是全局共享的 SQLite 文件（`meta.db`），**不存储任何业务数据**。职责：

1. 空间注册表（`spaces` 表）
2. 全局设置 KV（`meta_settings` 表，含主密码）

### 4.2 spaces 表

```python
# backend/app/models/meta.py（新增文件）

from sqlalchemy import Column, String, Boolean, Integer
from backend.app.models.base import Base

class Space(Base):
    __tablename__ = "spaces"

    id = Column(String, primary_key=True)           # UUID，如 "default"、"test-001"
    name = Column(String(100), nullable=False)        # 空间显示名称，如"日常"、"测试"
    icon = Column(String(10), default="📋")          # emoji 图标
    color = Column(String(7), default="#6366f1")     # 主题色 HEX，如 "#6366f1"
    db_path = Column(String(255), nullable=False)    # SQLite 文件路径，如 "space_default.db"
    password_hash = Column(String(255), nullable=True)  # 可选空间密码 (bcrypt)
    is_hidden = Column(Boolean, default=False)        # 隐私空间标记
    sync_enabled = Column(Boolean, default=True)      # 是否启用同步
    sort_order = Column(Integer, default=0)           # 排序
    created_at = Column(String(32), nullable=False)   # ISO 8601
    updated_at = Column(String(32), nullable=False)   # ISO 8601
```

**字段说明**：

| 字段 | 类型 | 用途 | 示例 |
|------|------|------|------|
| `id` | String (UUID) | 空间唯一标识 | `"default"`, `"a3f8b2c1"` |
| `name` | String(100) | 用户可见名称 | `"日常"`, `"测试"`, `"隐私"` |
| `icon` | String(10) | emoji 图标 | `"📋"`, `"🔒"`, `"🧪"` |
| `color` | String(7) | HEX 主题色 | `"#6366f1"`, `"#ef4444"` |
| `db_path` | String(255) | SQLite 文件名 | `"space_default.db"` |
| `password_hash` | String(255) | 空间密码 bcrypt 哈希 | `null`（无密码）或 `"$2b$12$..."` |
| `is_hidden` | Boolean | 隐私空间标记 | `false`（正常）/ `true`（隐藏） |
| `sync_enabled` | Boolean | 同步开关 | `true` / `false` |
| `sort_order` | Integer | 显示排序 | `0`, `1`, `2` |
| `created_at` / `updated_at` | String(32) | 时间戳 | `"2026-07-02T10:00:00.000Z"` |

### 4.3 meta_settings 表

```python
class MetaSetting(Base):
    __tablename__ = "meta_settings"

    key = Column(String(100), primary_key=True)      # 如 "admin_password", "theme"
    value = Column(String, nullable=True)              # 值（字符串，复杂结构用 JSON）
    updated_at = Column(String(32), nullable=False)
```

**存储内容**：

| key | value | 说明 |
|-----|-------|------|
| `admin_password` | bcrypt hash | 主密码（从原 `settings` 表迁移） |
| `theme` | `"dark"` / `"light"` | 全局主题（所有空间共享） |
| `language` | `"zh-CN"` | 全局语言 |
| `last_space_id` | `"default"` | 上次使用的空间（快速恢复） |

### 4.4 与业务 DB 的关系

```
meta.db（全局共享）
├── spaces 表          ← 空间注册表
└── meta_settings 表   ← 主密码 + 全局设置

space_default.db（"日常"空间）
├── tasks, sessions, reflections, ...  ← 16 张业务表
├── settings 表                           ← 空间级设置（专注时长、提醒规则等）
├── outbox, syncMeta                      ← 同步基础设施
└── tombstones                            ← 墓碑

space_test.db（"测试"空间）
├── tasks, sessions, reflections, ...  ← 完全独立的业务数据
├── settings 表                           ← 独立的空间级设置
├── outbox, syncMeta                      ← 独立的同步通道
└── tombstones                            ← 独立的墓碑
```

**设计原则**：
- `admin_password` 属于全局（`meta_settings`），不属于任何空间
- 空间级设置（如专注时长）属于各自空间的 `settings` 表
- 全局设置（如主题、语言）属于 `meta_settings`，所有空间共享

### 4.5 Meta Engine 初始化

```python
# backend/app/database.py（新增）

meta_engine = create_async_engine(
    "sqlite+aiosqlite:///./meta.db",
    echo=settings.debug
)

meta_async_session = async_sessionmaker(
    meta_engine, class_=AsyncSession, expire_on_commit=False
)

async def get_meta_db() -> AsyncSession:
    """Meta DB 依赖注入——用于空间管理、认证"""
    async with meta_async_session() as session:
        yield session

async def init_meta_db():
    """初始化 Meta DB——创建 spaces + meta_settings 表"""
    async with meta_engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all, tables=[
            Space.__table__, MetaSetting.__table__
        ])
```

## 五、业务数据库设计

### 5.1 核心原则：16 表 schema 完全不变

选择独立 DB 文件方案后，**每张业务表无需添加 `space_id` 列**。数据在独立 DB 文件中天然隔离，所有现有模型代码、路由查询、同步逻辑的 WHERE 子句都无需修改。

| 表名 | 说明 | 多空间变更 |
|------|------|-----------|
| `tasks` | 任务 | **无变更** |
| `sessions` | 番茄钟会话 | **无变更** |
| `reflections` | 复盘 | **无变更** |
| `settings` | 空间级设置 KV | **无变更**（每空间独立配置） |
| `tombstones` | 墓碑 | **无变更**（唯一约束不需扩展，DB 物理隔离） |
| `schedules` | 日程 | **无变更** |
| `quick_notes` | 速记 | **无变更** |
| `notes` | 笔记 | **无变更** |
| `folders` | 文件夹 | **无变更** |
| `habits` / `habit_check_ins` | 习惯 + 打卡 | **无变更** |
| `time_blocks` | 时间块 | **无变更** |
| `memo_comments` | 速记评论 | **无变更** |
| `session_quick_notes` | 会话-速记关联 | **无变更** |
| `schedule_quick_notes` | 日程-速记关联 | **无变更** |
| `task_quick_notes` | 任务-速记关联 | **无变更** |
| `outbox` / `syncMeta` | 同步基础设施 | **无变更**（每空间独立通道） |

### 5.2 独立 DB 文件方案的优势

| 对比维度 | space_id 列方案 | 独立 DB 文件方案（本设计） |
|---------|----------------|------------------------|
| 数据泄露风险 | 高（遗漏 WHERE 即泄露） | 零（物理隔离） |
| 现有代码改动 | 极高（74 处路由 + 所有查询） | 极低（仅替换依赖注入） |
| tombstone 唯一约束 | 需扩展为 `(space_id, entity_type, entity_id)` | 不变 |
| SQLite 写锁竞争 | 有（多空间共享 DB 写锁） | 无（独立文件） |
| 空间删除 | 逻辑删除（需标记 + 清理） | 物理删除（删除文件即可） |
| 备份/恢复 | 全量（包含所有空间） | 按空间独立备份/恢复 |
| 加密 | 整库加密 | 按空间独立加密（隐私空间可更强） |

### 5.3 新空间创建流程

```
POST /spaces (Master Token)
  │
  ├── 1. 生成 space_id (UUID)
  ├── 2. 在 meta.db.spaces 注册: id, name, db_path="space_{id}.db"

  ├── 5. 执行 _migrate_schema (索引创建等)
  ├── 6. 初始化系统文件夹 ("归档的小记" 等)
```

## 六、SpaceEngineManager

### 6.1 设计目标

SpaceEngineManager 管理多空间 SQLite 引擎池，负责：
- 按需创建/缓存 SQLAlchemy AsyncEngine
- LRU 策略淘汰长时间未使用的引擎
- 新空间 DB 的 schema 初始化

### 6.2 架构示意

```python
# backend/app/space_manager.py（新增文件）

import asyncio
from collections import OrderedDict
from sqlalchemy.ext.asyncio import create_async_engine, AsyncEngine, AsyncSession, async_sessionmaker
from backend.app.models.base import Base
from backend.app.database import _migrate_schema

class SpaceEngineManager:
    """管理多空间 SQLite 引擎池，LRU 淘汰"""

    def __init__(self, max_pool_size: int = 5):
        self._pool: OrderedDict[str, AsyncEngine] = OrderedDict()
        self._max_pool_size = max_pool_size
        self._lock = asyncio.Lock()  # 防止并发创建同一空间的引擎
        self._init_hooks = []  # 新空间初始化钩子

    async def get_engine(self, space_id: str, db_path: str) -> AsyncEngine:
        """获取或创建指定空间的数据库引擎"""
        async with self._lock:
            # 命中缓存
            if space_id in self._pool:
                engine = self._pool.pop(space_id)
                self._pool[space_id] = engine  # LRU: 移到末尾
                return engine

            # 创建新引擎
            engine = create_async_engine(
                f"sqlite+aiosqlite:///./{db_path}",
                echo=False
            )
            # 设置 SQLite PRAGMA (WAL, busy_timeout, synchronous)
            @event.listens_for(engine.sync_engine, "connect")
            def _set_pragma(dbapi_conn, _):
                cursor = dbapi_conn.cursor()
                cursor.execute("PRAGMA journal_mode=WAL")
                cursor.execute("PRAGMA busy_timeout=5000")
                cursor.execute("PRAGMA synchronous=NORMAL")
                cursor.close()

            # 初始化 schema（首次创建）
            async with engine.begin() as conn:
                await conn.run_sync(Base.metadata.create_all)
                # 执行 _migrate_schema 补齐索引
                await _migrate_schema(conn)

            # 执行初始化钩子（系统文件夹等）
            for hook in self._init_hooks:
                await hook(space_id, engine)

            # LRU 淘汰
            while len(self._pool) >= self._max_pool_size:
                _, old_engine = self._pool.popitem(last=False)
                await old_engine.dispose()

            self._pool[space_id] = engine
            return engine

    async def get_session(self, space_id: str, db_path: str) -> AsyncSession:
        """获取指定空间的数据库 session"""
        engine = await self.get_engine(space_id, db_path)
        session_factory = async_sessionmaker(engine, expire_on_commit=False)
        return session_factory()

    async def dispose(self, space_id: str):
        """关闭并移除指定空间的引擎"""
        engine = self._pool.pop(space_id, None)
        if engine:
            await engine.dispose()

    async def dispose_all(self):
        """关闭所有引擎（应用关闭时调用）"""
        for space_id, engine in list(self._pool.items()):
            await engine.dispose()
        self._pool.clear()

    def register_init_hook(self, hook):
        """注册新空间初始化钩子"""
        self._init_hooks.append(hook)


# 全局单例
space_manager = SpaceEngineManager(max_pool_size=5)
```

### 6.3 LRU 淘汰策略

```
引擎池状态 (max=5):

[space_A] → [space_B] → [space_C] → [space_D] → [space_E]
   ↑最旧                                              ↑最新

访问 space_C (命中):
[space_A] → [space_B] → [space_D] → [space_E] → [space_C]
                                               ↑移到末尾

访问 space_F (未命中, 池满):
[space_B] → [space_D] → [space_E] → [space_C] → [space_F]
  ↑space_A 被 dispose                              ↑新引擎
```

**为什么 max=5**：单用户场景下同时活跃的空间通常 ≤ 5。超过 5 个时，最久未使用的空间引擎被 dispose（释放文件句柄），下次访问时自动重建。

### 6.4 与现有 database.py 的关系

| 现有组件 | 多空间后 | 说明 |
|---------|---------|------|
| `engine`（全局单例） | 保留，作为 Meta Engine | Meta DB 仍用全局单例 |
| `async_session` | 保留，作为 Meta Session | Meta DB 操作使用 |
| `get_db()` | 保留，改为 `get_meta_db()` | 用于空间管理路由 |
| `_migrate_schema()` | 复用 | 新空间 DB 创建后调用 |
| `init_db()` | 扩展 | 新增 Meta DB 初始化 + 单空间迁移 |

## 十、前端 Dexie 多数据库管理

### 10.1 方案：动态实例 + Proxy 代理

**核心思路**：保持对外接口 `export const db` 不变，通过 Proxy 代理将所有属性访问转发到当前空间的 Dexie 实例。

```typescript
// frontend/src/services/space-db.ts（新增文件）

import Dexie from 'dexie'
import type { Table } from 'dexie'
import { useSpaceStore } from '@/stores/space'

/**
 * SpaceDBManager 管理多空间的 Dexie 实例
 * 通过 Proxy 代理对外暴露与单例 db 完全相同的接口
 */
class SpaceDBManager {
  private currentDB: PomodoroXIDB | null = null
  private currentSpaceId: string | null = null

  /**
   * 切换到指定空间的 Dexie DB
   */
  switchTo(spaceId: string): void {
    // 1. 关闭旧连接
    if (this.currentDB) {
      this.currentDB.close()
    }
    // 2. 打开新空间 DB
    this.currentDB = new PomodoroXIDB(`pomodoroxi_${spaceId}`)
    this.currentSpaceId = spaceId
    // 3. 触发 store 重载（由各 store 监听 space 变化）
    window.dispatchEvent(new CustomEvent('space-switched', {
      detail: { spaceId }
    }))
  }

  /**
   * 获取当前 Dexie 实例
   */
  get current(): PomodoroXIDB {
    if (!this.currentDB) {
      throw new Error('No space selected. Call switchTo() first.')
    }
    return this.currentDB
  }

  /**
   * 创建 Proxy 代理——将所有属性访问转发到 currentDB
   * 业务代码 db.tasks.put() → currentDB.tasks.put()
   */
  get proxy(): PomodoroXIDB {
    const manager = this
    return new Proxy({} as PomodoroXIDB, {
      get(_target, prop: string) {
        const db = manager.current
        const value = (db as any)[prop]
        return typeof value === 'function' ? value.bind(db) : value
      }
    })
  }
}

// 全局单例
const spaceDBManager = new SpaceDBManager()

// 导出 Proxy 实例，替换原 export const db = new PomodoroXIDB('pomodoroxi')
export const db = spaceDBManager.proxy as PomodoroXIDB

// 导出管理器实例供 SpaceStore 使用
export { spaceDBManager }
```

```typescript
  async (error) => {
    if (error.response?.status === 401) {

      const masterToken = useSpaceStore.getState().masterToken
      const currentSpaceId = useSpaceStore.getState().currentSpaceId
      if (masterToken && currentSpaceId) {

          error.config.headers.Authorization = `Bearer ${res.data.space_token}`
          return api.request(error.config)

          // Master Token 也过期，跳转登录

        }
```

## 十三、数据迁移策略

### 13.1 迁移触发条件

在 `init_db()` 中检测：如果 `meta.db` 不存在但 `pomodoroxi.db` 存在，执行单空间 → 多空间迁移。

### 13.2 后端迁移（自动执行）

```python
# backend/app/database.py（init_db 中新增）

async def init_db():
    # ... 现有初始化逻辑

    # 检测是否需要单空间→多空间迁移
    if not Path("meta.db").exists() and Path("pomodoroxi.db").exists():
        await _migrate_single_to_multi_space()


async def _migrate_single_to_multi_space():
    """单空间 → 多空间迁移（幂等，失败可重试）"""
    try:
        # 1. 备份
        shutil.copy("pomodoroxi.db", "pomodoroxi.db.bak")

        # 2. 创建 Meta DB
        async with meta_engine.begin() as conn:
            await conn.run_sync(Base.metadata.create_all, tables=[
                Space.__table__, MetaSetting.__table__
            ])

        # 3. 迁移 admin_password
        async with async_session() as old_db:
            result = await old_db.execute(
                select(Setting).where(Setting.key == "admin_password")
            )
            admin_setting = result.scalar_one_or_none()
            if admin_setting:
                async with meta_async_session() as meta_db:
                    meta_db.add(MetaSetting(
                        key="admin_password",
                        value=admin_setting.value,
                        updated_at=utc_now_iso()
                    ))
                    # 从旧 DB 删除（避免重复）
                    await old_db.delete(admin_setting)
                    await meta_db.commit()
                    await old_db.commit()

        # 4. 重命名 DB 文件
        os.rename("pomodoroxi.db", "space_default.db")

        # 5. 在 Meta DB 注册默认空间
        async with meta_async_session() as meta_db:
            meta_db.add(Space(
                id="default",
                name="日常",
                icon="📋",
                db_path="space_default.db",
                sync_enabled=True,
                created_at=utc_now_iso(),
                updated_at=utc_now_iso()
            ))
            await meta_db.commit()

        logger.info("Single-space → multi-space migration completed successfully")
    except Exception as e:
        logger.error(f"Migration failed: {e}")
        # 恢复备份
        if Path("pomodoroxi.db.bak").exists() and not Path("pomodoroxi.db").exists():
            os.rename("pomodoroxi.db.bak", "pomodoroxi.db")
        raise
```

### 13.3 前端迁移（首次启动检测）

```typescript
// frontend/src/services/space-db.ts（新增迁移逻辑）

async function checkAndMigrateFrontend(): Promise<void> {
  // 检测旧 Dexie DB 是否存在
  const oldDBNames = await Dexie.getDatabaseNames()
  if (oldDBNames.includes('pomodoroxi') && !oldDBNames.includes('pomodoroxi_default')) {
    // 执行迁移：导出旧 DB 数据 → 导入新 DB
    await migrateDexieData('pomodoroxi', 'pomodoroxi_default')
  }
}

async function migrateDexieData(fromDB: string, toDB: string): Promise<void> {
  const oldDB = new PomodoroXIDB(fromDB)
  const newDB = new PomodoroXIDB(toDB)

  // 逐表导出/导入
  const tableNames = oldDB.tables.map(t => t.name)
  for (const tableName of tableNames) {
    const allRecords = await oldDB.table(tableName).toArray()
    if (allRecords.length > 0) {
      await newDB.table(tableName).bulkPut(allRecords)
    }
  }

  oldDB.close()
  // 删除旧 DB
  await Dexie.delete(fromDB)
  logger.info(`Migrated Dexie data: ${fromDB} → ${toDB}`)
}
```

### 13.4 迁移安全性

| 安全措施 | 说明 |
|---------|------|
| 后端备份 | `pomodoroxi.db.bak` 保留 7 天 |
| 后端事务 | 迁移在事务中执行，失败则回滚 + 恢复备份 |

## 十七、风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| Dexie 配额不足 | 低 | 中 | `navigator.storage.persist()` + 配额监控 + LRU 空间数据清理 |
| 引擎池并发冲突 | 低 | 中 | `asyncio.Lock` 防止并发创建同一空间引擎 |
| 空间切换时数据残留 | 中 | 高 | Proxy 代理确保所有访问指向当前 DB；store 重载清空旧数据 |
| 迁移失败导致数据丢失 | 低 | 高 | 备份 .bak 文件 + 事务回滚 + 幂等检测 |
| Space Token 过期导致请求中断 | 中 | 低 | Axios 401 拦截器自动用 Master Token 刷新 |
| 隐私空间被意外显示 | 低 | 高 | API 层 `is_hidden` 过滤 + 前端需主密码解锁 |
