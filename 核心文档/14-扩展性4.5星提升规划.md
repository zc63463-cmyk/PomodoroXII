# PomodoroXII 扩展性 4.5 星提升规划

## 1. 目标与判断

目标不是把所有扩展点做成“零代码自动化”，而是把每类扩展稳定提升到 4.5 星：扩展步骤少、边界清晰、不会悄悄漏接入，并且遗漏能被测试或脚手架立即发现。

当前架构已经具备较好的基础：业务 Service 不依赖 FastAPI，SyncService 有通用 push/pull/status，实体元数据通过 registry 暴露给 REST/MCP，MCP 工具有 FastMCP 装饰器模式。但上一轮核查发现，当前仍有多个手工同步点：

- `registry/builtin.py` 与 `services/sync.py::ENTITY_REGISTRY` 是两套事实源。
- 新实体需要同步模型导入、Pydantic schema、路由聚合、迁移、测试门禁。
- `trash.py` 的 soft-delete 实体映射是硬编码，不是 registry 驱动。
- Stats 已出现 REST 与 MCP 不一致：REST 有 `daily-detail`，MCP 无对应工具。
- MCP 代码依赖 `fastmcp`，但当前测试环境未安装该依赖，MCP server 测试不能通过。
- 多个测试写死 20 个实体、14 个 sync 实体，新增实体时会人工维护。

提升原则：

1. 单一事实源优先：实体能力、sync 名称、pull key、删除策略、路由暴露策略应尽量落在同一个结构化 spec。
2. 允许显式代码，不追求黑盒魔法：简单 CRUD 可脚手架生成，复杂实体继续手写，但必须有一致性检查。
3. 所有自动传播都要有门禁：registry、sync、trash、OpenAPI、MCP、迁移、测试都要能发现遗漏。
4. 保持现有 API 兼容：尤其是 sync entity_type 的 camelCase 历史协议，如 `quickNote`、`timeBlock`。

## 2. 目标评分

| 维度 | 当前主要短板 | 4.5 星目标 |
|---|---|---|
| 加新实体 | 事实源分裂，步骤漏算，迁移/测试手工 | 一个 EntitySpec + 脚手架生成常规文件，sync/trash/MCP/status 一致性自动检查 |
| 加新字段 | 漏迁移风险，model/schema/registry 可能漂移 | 字段三处仍可显式维护，但有 schema/model/registry/migration parity gate |
| 加 MCP 工具 | 装饰器轻，但环境测试失败 | `fastmcp` 环境固定，工具注册/执行测试可稳定运行 |
| 加统计维度 | Service/Route/MCP 容易漂移 | StatSpec 或装饰器统一登记，至少有 REST/MCP parity test |
| 加新路由组 | `routes/v1/__init__.py` 手工 include | 路由清单集中化，脚手架或测试发现未 include |
| 加同步安全检查 | `_check_preflight` 通用，但 entity-specific hook 分散 | 通用 preflight + EntitySpec hooks，特殊检查可注册、可测试 |

## 3. 架构提升方向

### 3.1 扩展 EntitySpec，形成实体能力事实源

在现有 `EntitySpec` 基础上增加扩展元数据，建议不要一次全自动改完，而是先增加字段并让旧路径兼容：

```python
@dataclass(frozen=True)
class EntitySpec:
    name: str
    model_path: str
    table_name: str
    storage_type: StorageType
    category: EntityCategory
    sync_enabled: bool
    soft_delete: bool
    fields: tuple[FieldSpec, ...]
    primary_key: str = "id"
    description: str = ""

    # 新增建议
    sync_entity_type: str | None = None      # e.g. quickNote
    pull_key: str | None = None              # e.g. quickNotes
    route_prefix: str | None = None          # e.g. quick-notes
    service_path: str | None = None          # app.services.quick_note.QuickNoteService
    schema_module: str | None = None         # app.schemas.quick_note
    schema_prefix: str | None = None         # QuickNote
    delete_strategy: str = "hard_tombstone"  # hard_tombstone|soft_delete|cascade_soft_delete|fs_saga|none
    route_enabled: bool = False
    mcp_schema_enabled: bool = True
```

技术要求：

- `sync_entity_type` 不填时默认使用 `name`，但现有 camelCase 实体必须显式声明，保证协议兼容。
- `pull_key` 不填时可由 `sync_entity_type` 复数化生成，但现有实体应先显式声明，避免复数规则误伤。
- `delete_strategy` 必须驱动 REST delete、trash restore/purge、tombstone 生成策略的检查。
- `model_path` 需要配套 `resolve_model(spec)`，统一 import ORM class。

### 3.2 从 registry 派生 SyncService registry

目标：逐步替代 `services/sync.py::ENTITY_REGISTRY` 的手工维护。

建议实现：

```python
def build_sync_registry() -> dict[str, dict[str, Any]]:
    result = {}
    for spec in REGISTRY.list_sync_enabled():
        model = resolve_model(spec.model_path)
        entity_type = spec.sync_entity_type or spec.name
        pull_key = spec.pull_key or default_pull_key(entity_type)
        result[entity_type] = {"model": model, "pull_key": pull_key, "spec": spec}
    return result
```

技术要求：

- 第一阶段保留 `ENTITY_REGISTRY` 名称，内部改为调用 `build_sync_registry()`，减少调用方改动。
- 保留 `note` 的 FS+DB 特殊路径，新增 FS+DB 实体不得默认走 ORM 通用路径。
- 增加测试：`ENTITY_REGISTRY` 与 `REGISTRY.list_sync_enabled()` 数量、entity_type、pull_key 完全一致。
- 增加协议兼容测试：`quickNote/timeBlock/habitCheckIn` 等历史 sync entity_type 不改变。

### 3.3 统一模型加载与迁移约束

新增 ORM 模型必须被 `Base.metadata` 看见，否则 `create_all`、Alembic autogenerate、测试 metadata 都可能漏表。

建议实现：

- 新增 `app.registry.resolve.import_all_models()`，按 `REGISTRY.list()` 的 `model_path` import 所有 ORM class。
- `app.models.__init__` 可以继续显式 re-export，但 Alembic 与 space schema init 应调用 `import_all_models()`。
- `alembic/env.py` 在设置 `target_metadata` 前确保所有模型已导入。
- `space_manager._init_schema()` 在 `Base.metadata.create_all` 前确保模型已导入。

技术要求：

- 生产路径仍以 Alembic 迁移为准，`create_all` 只用于测试/新空间兜底。
- 新实体和新增字段必须有 migration，不能只改 ORM。
- 增加 `test_alembic_metadata_parity`：Alembic head 生成的表/列应覆盖当前 metadata。

### 3.4 提供实体脚手架，而不是完全动态 CRUD

4.5 星最稳妥的路线是脚手架生成显式代码。完全动态 CRUD 会牺牲可读性，也不适合 Note、Folder、Relation 这类特殊实体。

建议新增：

```text
backend/scripts/scaffold_entity.py
```

输入：

```text
name=goal
class_name=Goal
table_name=goals
route_prefix=goals
sync_entity_type=goal
pull_key=goals
delete_strategy=hard_tombstone
fields=...
```

输出常规文件：

- `app/models/goal.py`
- `app/schemas/goal.py`
- `app/services/goal.py`
- `app/routes/v1/goals.py`
- `app/registry/builtin.py` 中的 EntitySpec 草案或单独 registry fragment
- `app/routes/v1/__init__.py` 的 import/include 草案
- Alembic revision 草案
- 对应基础测试草案

技术要求：

- 脚手架生成的代码必须是普通可读 Python，不依赖运行时黑盒。
- 生成后必须运行格式化和测试。
- 特殊实体允许 `--custom-service`、`--soft-delete`、`--fs-db-split` 标记，并要求人工补特殊逻辑。

### 3.5 Registry 驱动 trash 能力

当前 trash 只认识硬编码的 `note/folder/quick_note`。提升目标是 soft-delete 实体由 registry 发现，特殊 cascade 逻辑由 delete strategy 决定。

建议实现：

- `_ENTITY_MAP` 改为由 `REGISTRY.list_soft_delete()` + `resolve_model()` 构建。
- `TrashService` 从路由中抽出，封装 list/restore/purge。
- `delete_strategy` 决定：
  - `hard_tombstone`：REST delete 物理删除并写 tombstone。
  - `soft_delete`：REST delete 设置 `trashed_at`，pull 通过 updated_at 同步状态；purge 再写 tombstone。
  - `cascade_soft_delete`：Folder 类实体走 cascade。
  - `fs_saga`：Note 类实体保持 FS/DB saga。

技术要求：

- 软删除不等于 tombstone。只有物理删除或 purge 才写 tombstone。
- REST delete、trash restore、trash purge 的行为必须有 integration test。
- 新 soft-delete 实体只要 EntitySpec 正确，trash list/restore/purge 应自动覆盖。

### 3.6 字段扩展 parity gate

新增字段仍建议显式改三处：ORM model、Pydantic schema、EntitySpec。为了达到 4.5 星，关键是让遗漏必失败。

建议新增检查：

```text
tests/test_entity_parity.py
```

检查内容：

- 每个 registry field 必须存在于 ORM `__table__.columns`，允许声明例外。
- 每个 ORM 业务列必须存在于 registry fields，允许声明内部列例外。
- Create/Update/Response schema 与 registry 的可写/只读字段匹配，允许例外，如 Note 的 `content`。
- 新增非 nullable 字段必须有 ORM default、server_default 或 migration backfill 说明。

技术要求：

- `FieldSpec` 可扩展 `readable/writable/syncable/api_visible`，减少 schema 例外散落。
- JSON 字符串字段，如 `tags/rest_days`，需要声明序列化策略，避免 sync/API 表现不一致。

### 3.7 Stats 维度统一登记

当前 REST stats 与 MCP stats 已有漂移。建议引入轻量 StatSpec，而不是继续三处手写后靠记忆同步。

建议实现：

```python
@stat_dimension(
    name="goal_summary",
    route_path="/goal-summary",
    mcp_tool="get_goal_summary",
    params={"days": {"default": 30, "ge": 1, "le": 365}},
)
async def goal_summary(self, days: int = 30) -> dict:
    ...
```

落地可以分两步：

1. 先不动态生成路由/MCP，只建立 `STAT_SPECS` 并加 parity test。
2. 再把 `routes/v1/stats.py` 和 `mcp/server.py` 的 wrapper 生成或集中化。

技术要求：

- `daily_detail` 要么补 MCP 工具，要么在 StatSpec 中标记 `mcp_enabled=False` 并写明原因。
- 每个 stats 方法必须有 service unit test、REST route test、MCP tool test 或显式豁免。
- MCP 与 REST 参数约束必须一致。

### 3.8 MCP 工具与环境门禁

MCP 新工具本身保持装饰器模式即可，不建议过度抽象。但必须修复环境和测试门禁。

技术要求：

- 确认 `uv.lock`/开发环境包含 `fastmcp`。
- CI 或本地标准命令必须能通过 `tests/test_mcp_server.py`。
- 新 MCP tool 必须至少有一个 import/registration test；涉及 DB 的 tool 需要执行测试。
- MCP tool docstring 必须说明参数、返回结构和 space_id 约定。

### 3.9 路由聚合集中化

`routes/v1/__init__.py` 的手工 include 是清晰的，但容易漏。建议引入集中路由表：

```python
ROUTE_MODULES = [
    ("app.routes.v1.tasks", "tasks", "/tasks"),
    ...
]
```

`build_v1_router()` 遍历路由表 include。脚手架新增实体时只改路由表。

技术要求：

- 保留明确 tags 和 prefix。
- 增加测试：存在于 registry 且 `route_enabled=True` 的实体必须有对应 route module/include。
- OpenAPI 测试检查新 route path 存在。

### 3.10 同步安全检查 hook 化

当前 `_check_preflight` 已统一时间、client field stripping、tombstone 防复活。Folder circular ref 是 entity-specific 逻辑，未来类似检查会继续增加。

建议新增：

```python
class SyncPreflightHook(Protocol):
    async def __call__(self, db, etype, eid, action, payload, client_ts) -> str | None:
        ...
```

EntitySpec 可声明：

```python
preflight_hooks=("app.services.sync_hooks.folder_circular_ref",)
```

技术要求：

- 通用 hook 执行顺序固定：timestamp normalize -> strip client fields -> tombstone check -> entity hooks。
- hook 返回标准 conflict code。
- 每个 hook 必须有单元测试和至少一个 push 集成测试。

## 4. 分阶段实施路线

### Phase 1：一致性门禁优先

目标：先让遗漏暴露，不急着自动生成。

交付：

- `tests/test_entity_parity.py`
- registry vs sync parity test
- route include parity test
- stats REST/MCP parity test
- MCP 环境修复，`test_mcp_server.py` 可运行
- 文档化当前允许的例外：Note content、Folder cascade、Relation junctions

### Phase 2：registry 派生 sync/trash

目标：消除最危险的事实源分裂。

交付：

- EntitySpec 增加 `sync_entity_type/pull_key/delete_strategy`
- `build_sync_registry()`
- `TrashService` 与 registry-driven soft-delete map
- 保持现有 sync 协议兼容

### Phase 3：脚手架与迁移流程

目标：让新实体/字段扩展成为可重复流程。

交付：

- `scripts/scaffold_entity.py`
- `scripts/check_entity_consistency.py`
- Alembic metadata parity gate
- 以 `goal` 做一次 dry-run spike，证明新增实体扩展步骤可控

### Phase 4：Stats/MCP 注册化

目标：消除 stats 维度漂移。

交付：

- `StatSpec` 或 `@stat_dimension`
- REST/MCP wrapper 统一生成或集中注册
- 补齐 `daily_detail` 的 MCP 暴露或显式豁免

## 5. 验收标准

所有维度达到 4.5 星时，应满足：

1. 新增一个简单 DB-only sync 实体时，除业务字段定义外，不需要人工理解 sync/status/MCP schema/trash 的内部细节。
2. 新实体遗漏 registry、sync、route、schema、migration 中任一环节，测试必须失败并指出原因。
3. 新增字段遗漏 model/schema/registry/migration 任一关键环节，测试或检查脚本必须失败。
4. REST stats 与 MCP stats 没有无说明漂移。
5. `tests/test_mcp_server.py` 在标准开发环境中可通过。
6. 现有 sync entity_type 与 pull_key 不发生破坏性变化。
7. 文档给出特殊实体豁免表，避免自动化误伤 Note、Folder、Relation。

建议验证命令：

```powershell
cd backend
python -m pytest tests/test_registry.py tests/test_sync_service.py tests/test_mcp_server.py tests/test_alembic.py tests/test_models.py tests/test_schemas.py -q
```

如果使用 `uv` 管理依赖，应优先固化为：

```powershell
cd backend
uv run pytest tests/test_registry.py tests/test_sync_service.py tests/test_mcp_server.py tests/test_alembic.py tests/test_models.py tests/test_schemas.py -q
```

## 7. Agent 拆分建议

如果要并行派发，建议拆成三路：

1. Registry/Sync/Trash agent：负责 EntitySpec 扩展、build_sync_registry、delete_strategy、trash registry 化。
2. Stats/MCP agent：负责 StatSpec、daily_detail 漂移修复、MCP 依赖与测试环境。
3. Tooling/Test agent：负责 parity tests、scaffold_entity.py 设计、Alembic metadata parity。

三路 agent 必须共享同一个约束：先做一致性门禁，再做自动生成。没有门禁的自动化会把遗漏隐藏得更深。
