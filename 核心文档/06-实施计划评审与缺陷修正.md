## 总体评价

**方案可行，方向正确，但存在 3 个必须修复的严重缺陷后才能实施。**

- 10 个 Task 拆分合理，依赖顺序正确

## 一、严重缺陷（必须修复）

### 🔴 缺陷 1：Saga 的 db.commit() 击穿 SAVEPOINT 隔离

**位置**：Task 5 `NoteFileService` + Task 6 `note_sync_adapter`

**问题**：

现有 sync_push 的事务模型（`sync.py:575`）：

```python
for idx, event in enumerate(events):
    async with db.begin_nested():      # 每事件独立 SAVEPOINT
        # 处理事件...
    if event_applied:
        applied.append(idx)

await db.commit()                       # 循环外统一提交
```

但计划中的代码在事件处理内部调用了 `await self.db.commit()`：

```python
# NoteFileService.create_note (Task 5)
await self.db.flush()
await self.fs.create_note(...)
await self.db.commit()   # ← 提交整个 session！

# _handle_note_event (Task 6)
existing.updated_at = updated_at
await db.commit()        # ← 同样提交整个 session！
```

**后果**：如果 10 个事件中第 3 个失败，前 2 个已被 `commit()` 固化，无法回滚。SAVEPOINT 机制彻底失效，`applied` 列表与实际持久化状态不一致。

**修正方案**：

NoteFileService 不应自行 commit，改为只 flush，由调用方控制提交：

```python
class NoteFileService:
    async def create_note(self, ...) -> Note:
        note = Note(...)
        self.db.add(note)
        await self.db.flush()          # 写入但不提交

        try:
            await self.fs.create_note(...)
        except Exception:
            await self.db.rollback()   # 回滚 flush
            raise
        # 不 commit — 由 sync_push 统一提交

        await self.db.flush()          # 确保 fs 成功后 flush 一致
        return note

    async def update_note(self, ...) -> Note:
        note = await self.db.get(Note, note_id)
        # ... 更新字段 ...
        await self.db.flush()          # 只 flush
        return note                    # 不 commit
```

sync adapter 也不应 commit：

```python
async def _handle_note_event(db, fs, event, client_time):
    # ... 处理逻辑 ...
    # 不 commit — 在 sync_push 的 SAVEPOINT 内，由外层统一提交
    return {"ok": True}
```

### 🔴 缺陷 2：sync adapter 丢失 5 道安全防线

**位置**：Task 6 `note_sync_adapter.py`

现有 sync_push 对每个实体事件执行了 5 道安全检查，计划的 adapter 几乎全部跳过：

| 安全检查 | 现有 sync_push | 计划 adapter | 后果 |
|---------|---------------|-------------|------|
| 墓碑优先检查（防复活） | `sync.py:476-510` | ❌ 缺失 | 已删除的 note 被 push 复活 |
| 零点时间戳检测 | `sync.py:426-438, 589` | ❌ 缺失 | 外部脚本注入的零点时间戳破坏 LWW |
| 客户端字段剥离 | `sync.py:442-458` | ❌ 缺失 | `_dirty`/`_etag` 写入 ORM 报 TypeError |
| 文件夹循环引用检测 | `sync.py:545-567` | ❌ 缺失 | folder parent 指向自己/后代 → 死循环 |
| delete 创建 Tombstone | `sync.py:817-835` | ❌ 缺失 | 删除不可同步，其他设备无法感知 |

**修正方案**：

`_handle_note_event` 和 `_handle_folder_event` 必须复用 sync_push 的安全检查逻辑。建议提取为公共函数：

```python
# app/services/sync_safety.py（新建）
async def check_tombstone_first(db, entity_type, entity_id) -> dict | None:
    """墓碑优先检查 — 返回 conflict 或 None"""
    tomb = await db.execute(
        select(Tombstone).where(
            Tombstone.entity_type == entity_type,
            Tombstone.entity_id == entity_id,
        )
    )
    if tomb.scalar_one_or_none():
        return {"conflict": {"reason": "tombstone_exists"}}
    return None

def strip_client_fields(entity_data: dict) -> dict:
    """剥离客户端独有字段"""
    for key in ("synced", "_dirty", "_etag", "actual_pomodoros",
                "archive_file_path", "migrated_to_note_id"):
        entity_data.pop(key, None)
    return entity_data

def detect_zero_time(ts: str | None) -> bool:
    """零点时间戳检测"""
    if not ts:
        return False
    time_part = ts.split("T")[-1] if "T" in ts else ts
    return time_part.startswith("00:00:00")

async def check_folder_circular_ref(db, folder_id, new_parent_id) -> bool:
    """文件夹循环引用检测 — 返回 True 表示有环"""
    if folder_id == new_parent_id:
        return True
    # BFS 检查 new_parent_id 是否是 folder_id 的后代
    queue = [new_parent_id]
    while queue:
        current = queue.pop()
        children = await db.execute(
            select(Folder.id).where(Folder.parent_id == current)
        )
        for (child_id,) in children.all():
            if child_id == folder_id:
                return True
            queue.append(child_id)
    return False
```

然后在 adapter 中调用：

```python
async def _handle_note_event(db, fs, event, client_time):
    entity = event.get("entity", {})
    entity = strip_client_fields(entity)  # 必须先剥离

    if action in ("create", "update"):
        tomb_check = await check_tombstone_first(db, "note", entity_id)
        if tomb_check:
            return tomb_check

    # ... 后续处理 ...

    if action == "delete":
        svc = NoteFileService(db, fs)
        await svc.delete_note(entity_id)
        # 创建 tombstone！
        db.add(Tombstone(entity_type="note", entity_id=entity_id, deleted_at=utc_now_iso()))
        return {"ok": True}
```

### 🔴 缺陷 3：pull 注入 content 用 N+1 查询

**位置**：Task 6 Step 3

**问题**：

```python
# 计划代码 — N+1 查询
for note_dict, content in zip(note_rows, contents):
    note_dict["content"] = content or ""
    orm_note = await db.get(Note, note_dict["id"])  # ← 每条 note 一次查询！
    note_dict["content_hash"] = orm_note.content_hash if orm_note else ""
```

如果 pull 返回 50 条 note，就是 50 次额外 DB 查询，完全抵消了 `read_notes_batch` 的优化。

**修正方案**：content_hash 应在 ORM 查询时就一并取出：

```python
# 在 sync_pull 构建 notes 结果时，content_hash 已在 ORM 查询结果中
# 不需要额外查询
note_rows = result["changes"]["notes"]
# note_rows 已经是从 ORM 查询出来的 dict，content_hash 已经在内
# 只需批量读 .md 补充 content
note_ids = [n["id"] for n in note_rows]
contents = await fs.read_notes_batch(note_ids)
for note_dict, content in zip(note_rows, contents):
    note_dict["content"] = content or ""
    # content_hash 已经在 note_dict 中（从 ORM 查询得到）
    # 无需额外查询
```

如果 sync_pull 当前不返回 content_hash（因为模型还没加这个字段），则需要在 pull 查询时显式包含：

```python
# sync_pull 中 note 查询
notes_result = await db.execute(
    select(Note).where(Note.updated_at > since).limit(limit + 1)
)
for note in notes_result.scalars():
    note_dict = {
        "id": note.id,
        "title": note.title,
        "content_hash": note.content_hash,  # 直接从 ORM 对象取
        "updated_at": note.updated_at,
        # ... 其他字段
    }
    note_rows.append(note_dict)
```

## 二、中等缺陷（建议修复）

### 🟡 缺陷 4：update_note 声明备份 old_hash 但未使用

**位置**：Task 5 `NoteFileService.update_note`

```python
# 计划代码
old_hash = note.content_hash   # ← 声明备份
# ... 更新 ...
if new_content is not None:
    note.content_hash = _sha256(new_content)
# old_hash 从未用于回滚
```

如果 FS 的 `edit_note` 成功但后续 ORM 更新失败，.md 文件已是新内容但 DB 的 content_hash 还是旧值。

**修正**：FS 写入失败时用 old_hash 恢复，或改写入顺序（先 ORM flush 再 FS）：

```python
async def update_note(self, ...):
    note = await self.db.get(Note, note_id)
    old_hash = note.content_hash

    # 先更新 ORM（可回滚）
    if new_content is not None:
        note.content_hash = _sha256(new_content)
    # ... 更新其他字段 ...
    await self.db.flush()  # 不 commit

    # 再更新 FS（失败时回滚 ORM）
    try:
        if new_content is not None:
            await self.fs.edit_note(note_id, new_content)
        if folder_id is not None and folder_id != note.folder_id:
            await self.fs.move_note(note_id, folder_id)
        if title is not None or tags is not None:
            await self.fs.edit_note_meta(note_id, title=title, tags=tags)
    except Exception:
        note.content_hash = old_hash  # 恢复旧 hash
        await self.db.rollback()
        raise
```

## 三、轻微缺陷（可后续修正）

### 🟢 缺陷 6：NoteFileService.delete_note 是软删除但不创建 Tombstone

计划中 `delete_note` 设置 `trashed_at` 和 `status="trashed"`，但 sync adapter 的 delete 路径需要额外创建 Tombstone。两处逻辑分散，建议统一到 NoteFileService 中：

```python
async def delete_note(self, note_id: str, create_tombstone: bool = True):
    note = await self.db.get(Note, note_id)
    if not note:
        raise KeyError(f"Note {note_id} not found")
    await self.fs.delete_note(note_id)
    note.trashed_at = utc_now_iso()
    note.status = "trashed"
    if create_tombstone:
        self.db.add(Tombstone(
            entity_type="note", entity_id=note_id, deleted_at=utc_now_iso()
        ))
    await self.db.flush()
```

### 🟢 缺陷 7：content_hash 未加入 NoteBase/NoteUpdate

计划把 `content_hash` 只放在 `NoteResponse`，但 sync push 事件可能携带客户端的 `content_hash`（用于跳过未变化的正文下载）。应在 `NoteUpdate` 中加入 `content_hash: Optional[str]` 并在 adapter 中剥离（服务端权威计算）。

### 🟢 缺陷 8：folder delete 缺少 BFS 级联

计划中 `_handle_folder_event` 的 delete 路径只删除 folder 本身，不做 BFS 级联删除后代文件夹和清理 Note/QuickNote 的 folder_id 引用。现有 sync_push 对 folder delete 有完整的 BFS 级联（`sync.py:761-815`）。adapter 必须复用此逻辑。

## 五、修正后的实施建议

### 优先级排序

1. **先修缺陷 1+2**（Saga commit + 安全防线）— 这是阻塞实施的前提
2. **按计划执行 Task 1-4**（骨架/移植/扩展/模型改造）— 无阻塞
3. **修缺陷 3 后执行 Task 5**（NoteFileService）— 改为 flush 不 commit
4. **修缺陷 2+8 后执行 Task 6**（sync adapter）— 补全安全检查 + BFS 级联
5. **按计划执行 Task 7-10**（convert/notes API/一致性/验证）
6. **前端测试框架迁移**（`@vue/test-utils` → `@testing-library/react`）— 与后端 Task 并行，互不阻塞

### 建议新增 Task：提取 sync 安全检查公共函数

在 Task 5 和 Task 6 之间插入一个新 Task，将 sync_push 的安全检查逻辑提取为可复用函数，供 adapter 调用。这避免了 adapter 重新实现一遍，也确保两条路径（ORM 直处理 + file_system 桥接）的安全检查一致。

### 建议新增 Task：数据迁移脚本

移除 `Note.content` 字段是不可逆操作（SQLite 不支持 DROP COLUMN）。需要：

1. 新增 `content_hash` 列（`_migrate_schema` 自动 ADD COLUMN）
2. 将现有 `Note.content` 的值写入 .md 文件（一次性迁移）
3. 计算 content_hash 填入新列
4. `content` 列保留但标记为 deprecated（SQLite 无法删除列）

```python
# backend/app/services/migrate_notes_to_fs.py（新建）
async def migrate_notes_to_files(db, fs):
    """一次性迁移：将 Note.content 写入 .md 文件"""
    result = await db.execute(select(Note).where(Note.content != ""))
    for note in result.scalars():
        if not note.content:
            continue
        await fs.create_note(
            title=note.title,
            content=note.content,
            folder_id=note.folder_id,
            external_id=note.id,
        )
        note.content_hash = _sha256(note.content)
    await db.commit()
```

### 建议新增 Task：前端测试框架迁移

**背景**：PomodoroXII 前端从 Vue 3 替换为 React 19，组件测试工具链需同步迁移。file_system 子系统的后端 pytest 测试保持不变（详见缺陷 5 补充说明），仅前端测试受影响。此 Task 与后端 file_system 实施 Task 完全解耦，可并行推进。

**迁移范围**：

| 测试层 | pomodoroxi (Vue) | PomodoroXII (React) | 迁移方式 |
|--------|-----------------|---------------------|---------|
| 组件单元测试 | `@vue/test-utils` + Vitest | `@testing-library/react` + Vitest | 重写（API 范式差异大） |
| 测试运行器 | Vitest | Vitest | 保持不变 |
| E2E 测试 | Playwright | Playwright | 直接复用（框架无关） |

**关键改动 1：`@vue/test-utils` → `@testing-library/react`**

测试范式从「组件挂载 + 实例断言」变为「用户行为驱动 + DOM 查询」，更贴近真实用户交互，天然鼓励通过可访问性角色（role/label）定位元素：

```tsx
// pomodoroxi (Vue) — @vue/test-utils
import { mount } from '@vue/test-utils'
import { describe, it, expect } from 'vitest'
import NoteEditor from '@/components/NoteEditor.vue'

describe('NoteEditor', () => {
  it('renders title', () => {
    const wrapper = mount(NoteEditor, { props: { title: 'Hello' } })
    expect(wrapper.text()).toContain('Hello')
  })
})
```

```tsx
// PomodoroXII (React) — @testing-library/react
import { render, screen } from '@testing-library/react'
import { describe, it, expect } from 'vitest'
import { NoteEditor } from '@/components/NoteEditor'

describe('NoteEditor', () => {
  it('renders title', () => {
    render(<NoteEditor title="Hello" />)
    expect(screen.getByText('Hello')).toBeInTheDocument()
  })
})
```

**关键改动 2：Vue 组件测试 → React 组件测试**

API 映射关系：

| Vue 测试操作 | React 测试操作 | 说明 |
|-------------|--------------|------|
| `mount(Component, { props })` | `render(<Component {...props} />)` | 组件挂载 |
| `shallowMount(Component)` | `render(<Component />)` + mock 子组件 | 浅渲染 |
| `wrapper.find(selector)` | `screen.getByRole()` / `getByText()` | 元素查找 |
| `wrapper.findAll(selector)` | `screen.getAllByRole()` | 批量查找 |
| `wrapper.emitted('click')` | `fireEvent.click(el)` / `userEvent.click(el)` | 事件触发 |
| `wrapper.setValue(value)` | `fireEvent.change(el, { target: { value } })` | 表单输入 |
| `wrapper.vm.count` | 直接调用 hook / 测试状态输出 | 实例属性 |
| `wrapper.setProps({ ... })` | `rerender({ ... })` | 属性更新 |
| `wrapper.unmount()` | `unmount()` | 卸载 |

**关键改动 3：Playwright E2E 直接复用**

Playwright 操作的是浏览器页面，与前端框架无关。`page.click()`、`page.fill()`、`page.locator()` 等选择器基于 DOM 结构，Vue 与 React 渲染出的 HTML 一致即可直接复用：

```typescript

test('create note via UI', async ({ page }) => {
```
