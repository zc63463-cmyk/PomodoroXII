## 一、背景与动机

### 1.1 用户原始诉求

> "P0,P1 设计了什么元数据基座吗? 是否便于后续的 api 设计, 为其他项目提供元数据信息的获取"

随后用户要求："基于两个 mcp 提供的信息, 拟在深度规划下, 制定方案和可行性分析"。

### 1.2 现状审查结论

| 维度 | 现状 | 问题 |
|------|------|------|
| ENTITY_REGISTRY | ❌ 不存在 | Grep 全仓无匹配 |
| 元数据 API 端点 | ❌ 不存在 | `/api/v1/meta/*` 未注册 |
| Schema 内省机制 | ❌ 不存在 | 无法在运行时查询字段定义 |
| `trash.py` 的 `_ENTITY_MAP` | ⚠️ 局部硬编码 | 仅 3 实体（note/folder/quick_note），散落在路由层 |
| `models/__init__.py` | ⚠️ 仅 import 注册 | 无元数据描述（存储类型、sync 能力、软删支持） |
| Phase C Sync 引擎 | ❌ 完全未实现 | `services/sync.py` / `routes/v1/sync.py` / `schemas/sync.py` 全部不存在 |

**核心结论**：当前项目**无任何元数据基座**。P0/P1 修复仅解决了数据一致性问题，未触及元数据层。Phase C Sync 引擎在文档中声称"Tasks 1-4 已完成"，但实际代码完全缺失——这与元数据基座缺失直接相关（无 registry 驱动，sync 无法泛化）。

### 1.3 为什么需要元数据基座

CBM 知识图谱揭示了 3 个关键场景需要统一的元数据层：

1. **Phase C Sync 引擎泛化**：12 个业务实体 + 3 个 sync 基础设施表需要不同的处理路径（FS+DB split vs DB-only），目前只能在 sync 服务里写 `if entity_type == "note"` 分支。
2. **Trash 路由的 DRY 重构**：`_ENTITY_MAP` 是 registry 的雏形，但被锁在路由层，无法被 sync 服务、stats 服务复用。
3. **外部项目元数据获取**：用户明确提到"为其他项目提供元数据信息的获取"，需要稳定的 HTTP API 暴露 schema。

### 1.4 设计目标（按优先级）

| 优先级 | 目标 | 衡量标准 |
|--------|------|----------|
| P0 | 不破坏现有 253 测试 | 全量回归通过 |
| P0 | 纯增量，不修改业务代码 | 仅新增文件 + `__init__.py` 注册 |
| P1 | 驱动 Phase C Sync | `SyncService` 通过 registry 分发 |
| P1 | 替换 `trash.py` 硬编码 | `_ENTITY_MAP` 改为查询 registry |
| P2 | 暴露 HTTP 元数据 API | `/api/v1/meta/*` 4 个端点 |
| P2 | 提供 schema 内省 | 字段类型/约束/索引可查询 |
| P3 | 支持外部项目集成 | master token 即可消费 |

## 二、CBM 知识图谱关键发现

### 2.1 架构聚类（Leiden 社区检测）

CBM 识别出 12 个 de-facto 模块，其中与元数据相关的高内聚簇：

| Cluster | 成员 | Cohesion | 代表节点 | 元数据相关性 |
|---------|------|----------|----------|--------------|
| #33 | 61 | 0.65 | `create`, `TaskService`, `TombstoneService`, `list`, `exists` | 高 — 服务层 CRUD 模式统一 |
| #20 | 44 | 0.84 | `get_engine`, `init_meta_db`, `dispose_all`, `space_session` | 高 — DB 层初始化逻辑 |
| #58 | 29 | 0.79 | `utc_now_iso`, `StatsService`, `Task`, `overview` | 中 — 时间/统计基座 |
| #74 | 25 | 0.58 | `NoteService`, `read_note`, `_make_fs` | 高 — FS+DB split 实体 |
| #89 | 23 | 0.68 | `soft_delete_folder`, `CascadeService`, `purge_item`, `NotFoundError` | 高 — 软删/级联语义 |

**洞察**：
- Cluster #33 与 #89 的高重叠表明服务层与软删语义紧耦合，registry 需要显式声明 `soft_delete` 能力。
- Cluster #74 独立存在说明 Note 的 FS+DB split 是一等公民，registry 必须区分 `storage_type`。
- 无任何 cluster 涉及"registry"或"entity metadata"，证实元数据基座确实缺失。

### 2.2 路由层结构

CBM 列出 20 个路由签名，其中 `POST /{entity_type}/{entity_id}/restore`、`DELETE /{entity_type}/{entity_id}` 是**唯一的**泛型 entity_type 路由——这正是 registry 应该驱动的模式，但目前仅 trash.py 一处使用。

### 2.3 依赖边界

CBM 边类型主要为 `CALLS`，未发现 `IMPORTS-metadata` 类边。registry 模块的引入将创建新的依赖方向：

```
app.registry ← app.services.sync (Phase C)
app.registry ← app.routes.v1.trash (重构)
app.registry ← app.routes.v1.meta (新增)
```

registry 必须**不反向依赖** services/routes，保持纯数据层。

## 三、架构方案

### 3.1 方案对比

| 方案 | 描述 | 优点 | 缺点 | 推荐 |
|------|------|------|------|------|
| A | 在 `models/__init__.py` 加 `__entity_meta__` 字典 | 零新文件 | 模型层耦合元数据，违反"Models 纯数据"铁律 | ❌ |
| B | 在 `services/` 加 `registry.py` | 复用 services 包 | services 职责膨胀，与 BaseService 无关 | ❌ |
| **C** | 独立 `app/registry/` 包 | 纯数据层，可被任意层消费 | 多一个包 | ✅ |
| D | 注入到 `Base.metadata` | 复用 SQLAlchemy metadata | 污染 ORM metadata，与 sync 无关字段混杂 | ❌ |

**采用方案 C**：独立 `app/registry/` 包。理由：
1. 遵守三层铁律（registry 不导入 fastapi，不调用 commit）
2. 可被 services、routes、外部项目（通过 meta API）三方消费
3. 纯增量，零回归风险
4. 与 Phase C Sync 的 ENTITY_REGISTRY 设计完全对齐

### 3.3 核心数据类设计

```python
# app/registry/entities.py
from __future__ import annotations
from dataclasses import dataclass, field
from enum import Enum
from typing import Any


class StorageType(str, Enum):
    """实体存储模式。"""
    DB_ONLY = "db_only"           # 纯 DB 表（task, session, folder...）
    FS_DB_SPLIT = "fs_db_split"   # FS 存 content + DB 存 meta（仅 note）
    SYSTEM = "system"             # 系统表（tombstone, sync_outbox, sync_audit_log）


class EntityCategory(str, Enum):
    """实体业务分类。"""
    BUSINESS = "business"         # 业务实体（参与 sync）
    SYNC_INFRA = "sync_infra"     # sync 基础设施表
    META = "meta"                 # 元数据表（spaces, meta_settings）
    SETTING = "setting"           # 设置表（settings）


@dataclass(frozen=True)
class FieldSpec:
    """字段元数据。"""
    name: str
    type: str                     # string|integer|datetime|json|text|boolean
    nullable: bool
    default: Any = None
    indexed: bool = False
    unique: bool = False
    description: str = ""


@dataclass(frozen=True)
class EntitySpec:
    """实体元数据规范。"""
    name: str                     # entity_type，如 "note"
    model_path: str               # ORM 全限定名 "app.models.note.Note"
    table_name: str               # "notes"
    storage_type: StorageType
    category: EntityCategory
    sync_enabled: bool            # 是否参与 Phase C sync
    soft_delete: bool             # 是否支持 trashed_at 软删
    fields: tuple[FieldSpec, ...]
    primary_key: str = "id"
    description: str = ""

    @property
    def field_names(self) -> tuple[str, ...]:
        return tuple(f.name for f in self.fields)
```

### 3.4 Registry 单例设计

```python
# app/registry/__init__.py
from __future__ import annotations
from app.registry.entities import EntitySpec, StorageType, EntityCategory


class EntityRegistry:
    """实体元数据注册中心（进程内单例）。"""

    def __init__(self) -> None:
        self._specs: dict[str, EntitySpec] = {}

    def register(self, spec: EntitySpec) -> None:
        if spec.name in self._specs:
            raise ValueError(f"Entity {spec.name!r} already registered")
        self._specs[spec.name] = spec

    def get(self, name: str) -> EntitySpec:
        if name not in self._specs:
            raise KeyError(f"Unknown entity: {name!r}")
        return self._specs[name]

    def list(self) -> list[EntitySpec]:
        return list(self._specs.values())

    def list_by_category(self, cat: EntityCategory) -> list[EntitySpec]:
        return [s for s in self._specs.values() if s.category == cat]

    def list_sync_enabled(self) -> list[EntitySpec]:
        return [s for s in self._specs.values() if s.sync_enabled]

    def list_soft_delete(self) -> list[EntitySpec]:
        return [s for s in self._specs.values() if s.soft_delete]

    def __contains__(self, name: str) -> bool:
        return name in self._specs

    def __len__(self) -> int:
        return len(self._specs)


# 进程级单例
REGISTRY = EntityRegistry()

# 导入 builtin 触发注册
from app.registry import builtin  # noqa: E402, F401
```

### 4.3 路由实现

```python
# app/routes/v1/meta.py
from fastapi import APIRouter, Depends, Query
from app.deps import require_master_token
from app.registry import REGISTRY
from app.registry.entities import EntityCategory
from app.schemas.meta import EntityListOut, EntitySpecOut, RegistryHealthOut
from app.errors import NotFoundError, ValidationError

router = APIRouter()


@router.get("/health", response_model=RegistryHealthOut)
async def registry_health(user: dict = Depends(require_master_token)):
    cats = {}
    for s in REGISTRY.list():
        cats[s.category.value] = cats.get(s.category.value, 0) + 1
    return RegistryHealthOut(
        registry_loaded=len(REGISTRY) > 0,
        entity_count=len(REGISTRY),
        categories=cats,
    )


@router.get("/entities", response_model=EntityListOut)
async def list_entities(
    category: str | None = Query(None, description="Filter by category"),
    user: dict = Depends(require_master_token),
):
    if category is not None:
        try:
            cat = EntityCategory(category)
        except ValueError as e:
            raise ValidationError(f"Invalid category: {category!r}") from e
        specs = REGISTRY.list_by_category(cat)
    else:
        specs = REGISTRY.list()
    return EntityListOut(
        entities=[EntitySpecOut(**_serialize(s)) for s in specs],
        total=len(specs),
    )


@router.get("/entities/{entity_type}", response_model=EntitySpecOut)
async def get_entity(
    entity_type: str,
    user: dict = Depends(require_master_token),
):
    if entity_type not in REGISTRY:
        raise NotFoundError(f"Entity {entity_type!r} not found in registry")
    spec = REGISTRY.get(entity_type)
    return EntitySpecOut(**_serialize(spec))


@router.get("/entities/{entity_type}/schema")
async def get_entity_schema(
    entity_type: str,
    user: dict = Depends(require_master_token),
):
    """返回字段 schema（含约束），供外部代码生成器消费。"""
    if entity_type not in REGISTRY:
        raise NotFoundError(f"Entity {entity_type!r} not found")
    spec = REGISTRY.get(entity_type)
    return {
        "entity_type": spec.name,
        "table_name": spec.table_name,
        "primary_key": spec.primary_key,
        "fields": [_field_dict(f) for f in spec.fields],
    }


def _serialize(spec):
    """将 frozen dataclass 序列化为 dict（兼容 Pydantic）。"""
    return {
        "name": spec.name,
        "model_path": spec.model_path,
        "table_name": spec.table_name,
        "storage_type": spec.storage_type.value,
        "category": spec.category.value,
        "sync_enabled": spec.sync_enabled,
        "soft_delete": spec.soft_delete,
        "primary_key": spec.primary_key,
        "description": spec.description,
        "fields": [_field_dict(f) for f in spec.fields],
    }


def _field_dict(f):
    return {
        "name": f.name, "type": f.type, "nullable": f.nullable,
        "default": f.default, "indexed": f.indexed,
        "unique": f.unique, "description": f.description,
    }
```

### 5.1 SyncService 通过 registry 分发

```python
# app/services/sync.py （Phase C 实现时）
from app.registry import REGISTRY
from app.registry.entities import StorageType

class SyncService:
    def __init__(self, db, fs=None):
        self.db = db
        self.fs = fs

    async def push(self, events: list[dict]) -> dict:
        results = []
        for event in events:
            spec = REGISTRY.get(event["entity_type"])
            # 用 savepoint 隔离每个事件
            async with self.db.begin_nested():
                if spec.storage_type == StorageType.FS_DB_SPLIT:
                    # 委托给 NoteService（带 Saga 补偿）
                    result = await self._push_note(event)
                else:
                    # 直接 ORM 操作
                    result = await self._push_orm(spec, event)
                results.append(result)
        await self._write_audit("push", results)
        return {"pushed": len(results)}

    async def _push_note(self, event): ...
    async def _push_orm(self, spec, event): ...
```

## 七、风险评估与可行性分析

### 7.1 技术风险

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| dataclass + Enum 与 Pydantic v2 序列化冲突 | 中 | 中 | 用 `model_config = ConfigDict(from_attributes=True)` 或手动 `_serialize()` |
| registry 注册顺序导致 import 循环 | 低 | 高 | `builtin.py` 在 `__init__.py` 末尾导入；registry 不反向 import models |
| 字段描述与实际 ORM 漂移 | 高 | 低 | M-3 Gate 测试 + 未来可从 `Column` 反射自动生成 |
| Phase C Sync 实际实现与 registry 假设不符 | 中 | 中 | registry 设计已对齐 phase-c-sync-completion-plan.md 的 ENTITY_REGISTRY 规格 |
| Windows 路径大小写问题（历史教训） | 低 | 高 | 所有写入用绝对路径 `e:\Development\MyAwesomeApp\PomodoroXII\...` |

### 7.2 维护成本

| 维度 | 成本 | 说明 |
|------|------|------|
| 新增实体 | +5 行 | 在 `builtin.py` 加一个 `REGISTRY.register(EntitySpec(...))` |
| 字段变更 | +1 行 | 同步更新 `builtin.py` 中的 FieldSpec |
| Gate 测试 | 一次性 | M-3 后无需维护，自动捕获遗漏 |

**结论**：维护成本可接受。相比硬编码 `_ENTITY_MAP`，registry 的声明式注册更易审计。

### 7.3 可行性矩阵

| 维度 | 评级 | 依据 |
|------|------|------|
| 技术可行性 | ✅ 高 | 纯 Python dataclass + Pydantic，无新技术 |
| 架构兼容性 | ✅ 高 | 遵守三层铁律，纯增量 |
| 回归风险 | ✅ 低 | 仅新增文件 + 1 个 `__init__.py` 注册行 |
| 复用价值 | ✅ 高 | 驱动 Phase C + 重构 trash + 外部集成 |
| 工作量 | ⚠️ 中 | 7 个新文件 + 13 测试，预估单次会话可完成 |
| 性能影响 | ✅ 无 | registry 进程内单例，无 IO |

### 7.4 替代方案对比

| 方案 | 工作量 | 复用性 | 推荐 |
|------|--------|--------|------|
| **方案 C（独立 registry 包）** | 中 | 高 | ✅ |
| 不做 registry，Phase C 时局部硬编码 | 低 | 低 | ❌ 重复 trash.py 的反模式 |
| 用 SQLAlchemy `inspect()` 反射 | 高 | 中 | ⚠️ 无法表达 sync_enabled/soft_delete 等业务语义 |
| 生成 OpenAPI schema 作为元数据 | 高 | 中 | ⚠️ OpenAPI 是 API 层，不适合作为内部 registry |
