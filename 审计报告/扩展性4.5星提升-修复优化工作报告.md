## 1. 背景与目标

前期深度调研发现，项目在多实体扩展时存在多处"双重事实源"和"手工同步"隐患：

- `SyncService.ENTITY_REGISTRY` 与 `REGISTRY` 分别维护实体列表，新增实体需改两处
- `trash.py` 的 `_ENTITY_MAP` 硬编码 6 个可回收实体，无法随注册表自动扩展
- Stats 维度在 `StatsService` 方法名、REST 路由、MCP 工具名三处手工同步，已出现 `daily_detail` REST 与 MCP 漂移
- 缺少字段级 parity gate，模型/Schema/Registry/Alembic 容易不同步

**目标**：将"扩展性"维度从 3.5 星提升至 4.5 星，建立"Registry 单一事实源 + 派生注册 + Parity Gate + 统一登记"机制。

## 3. 详细工作内容

### 3.1 P1 基线加固（parity gate 奠基）

- **`backend/app/registry/entities.py`**：为 `EntitySpec` 新增 7 个字段
  - `sync_entity_type` / `pull_key`：同步协议兼容
  - `route_prefix` / `service_path` / `schema_module` / `schema_prefix`：未来脚手架使用
  - `delete_strategy` / `route_enabled` / `mcp_schema_enabled`：能力声明
  - 新增 2 个派生属性：`effective_sync_entity_type`、`effective_pull_key`
- **`backend/app/registry/builtin.py`**：为 14 个同步实体补全 `pull_key`；为 3 个实体标注 `delete_strategy`（`hard_tombstone` / `soft_delete` / `cascade_soft_delete`）
- **`backend/app/services/meta.py`** + **`backend/app/schemas/meta.py`**：`/meta/entities` 接口返回 `sync_entity_type`、`pull_key`
- **`backend/app/mcp/server.py`**：补齐 `get_daily_detail` 工具，修复 REST/MCP 漂移
- **测试**：新增 `test_entity_spec_extension.py`、`test_parity_registry_orm.py`、`test_parity_registry_schemas.py`、`test_parity_routes.py`、`test_parity_alembic_metadata.py`；`test_routes_meta.py` 新增 2 个断言
- **文档**：
  - `核心文档/14-扩展性4.5星提升规划.md`
  - `核心文档/15-扩展性parity例外表.md`（记录 7 类合法例外）

### 3.2 P2 派生注册（消除双重事实源）

- **`backend/app/registry/sync_registry.py`**：新增 `build_sync_registry()`，从 `REGISTRY.list_sync_enabled()` 派生同步注册表，统一处理：
  - DB-only 实体
  - FS+DB split 实体（Note）
  - camelCase ↔ snake_case 映射
- **`backend/app/services/sync.py`**：`ENTITY_REGISTRY = build_sync_registry()`，移除硬编码映射
- **`backend/app/routes/v1/trash.py`**：`_ENTITY_MAP` 改为从 `REGISTRY.list_soft_delete()` 派生
- **`backend/tests/test_parity_registry_sync.py`**：启用第 3 个测试，校验 `ENTITY_REGISTRY.pull_key == REGISTRY.effective_pull_key`
- **`backend/tests/test_build_sync_registry.py`**：验证派生结果覆盖全部 14 个同步实体

### 3.3 P3 工具链（扩展性工程化）

- **`backend/scripts/scaffold_entity.py`**：纯字符串拼接的实体脚手架 CLI（不引入 jinja2）
  - 支持 `--name / --class-name / --table-name / --route-prefix / --fields`
  - 输出 8 个文件块：model / schema / service / router / registry snippet / init / alembic / test
  - `dry-run` 模式打印到 stdout，非 dry-run 仅对新建文件写入
- **`backend/scripts/check_entity_consistency.py`**：CI parity gate
  - 自动收集并运行所有 `test_parity_*.py`
  - Windows 适配：用 `glob.glob(..., root_dir=backend_dir)` 显式展开 glob
- **测试**：`backend/tests/test_scaffold_entity.py` 覆盖 dry-run 输出与非法字段类型退出码
- **P3.3 Goal dry-run spike**：完整生成 Goal 实体 8 文件块，验证脚手架能力，未合并主代码库

### 3.4 P4 统一登记（Stats/MCP 一致性）

- **`backend/app/services/stats_spec.py`**：新增 `StatSpec` frozen dataclass + `STAT_SPECS`（7 维度）
  - 统一登记：`name` / `service_method` / `route_path` / `mcp_tool` / `params`
  - 替代原先三处手工同步
- **`backend/tests/test_stat_spec.py`**：TDD 驱动，验证 STAT_SPECS 覆盖全部 REST 端点、全部 MCP 工具、命名一致性
- **`backend/tests/test_parity_stats_mcp.py`**：重构为 STAT_SPECS 驱动，并保留 `daily_detail` MCP 存在性回归测试

### 3.5 相关底层加固（同轮顺带修复）

- `backend/app/services/sync_safety.py`：timestamp normalize + `sanitize_zero_time`
- `backend/app/models/mixins.py`：`updated_at` 加 `index=True`
- Alembic 新增 `backend/alembic/versions/006_sync_timestamp_normalize.py`
- NoteService sync_mode 保留客户端时间戳并跳过 tombstone 写入
