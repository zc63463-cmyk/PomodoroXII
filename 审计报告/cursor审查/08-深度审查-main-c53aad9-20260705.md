**REST 软删 vs sync 硬删差异**（文档化）：
- REST `DELETE /notes/{id}` = 软删（行保留 + `trashed_at` set）；`GET /notes/{id}` 仍返回 200 + `trashed_at` 字段
- REST `DELETE /trash/note/{id}` = purge 硬删（ORM delete + tombstone + fs.purge）
- Sync push delete = 直接硬删（`hard=True`，sync delete 是权威事件，不进 trash）

### §10.1 边缘行为约定（软删后 REST 行为）

> 本轮深度收尾 P3 勘探结论。仅 E-5 需新增守卫，其余项已正确实现，文档化备查。

| 场景 | 行为 | 实现位置 |
|------|------|---------|
| `GET /notes` 列表 | 不含软删项 | `notes.py` list_notes filter `trashed_at=None` |
| `GET /notes/search` | 不命中软删项 | `search_ops.py` SQL `is_deleted=0`（LIKE + FTS5 双路径） |
| `GET /notes/{id}` | 200 + `trashed_at` 非空 | `NoteService.get` 行存在 |
| `PUT /notes/{id}/content` | 404 | `fs.edit_note` `is_deleted=0` 守卫 → KeyError → 路由 404 |
| `PATCH /notes/{id}` | 422（需先 restore） | `NoteService.update_metadata` `trashed_at` 校验（本轮新增） |
| `GET /notes/{id}/versions` | 200（版本备份在 `.meta/`，不随 .md 移到 .trash） | `fs.list_versions` |
| `DELETE /trash/note/{id}` (purge) | 200 + tombstone | `trash.py` purge_item |
| `POST /trash/note/{id}/restore` 路径冲突 | 409 | `trash.py` restore_item `FileExistsError` → ConflictError（本轮新增） |
